# 客户资产批量治理与导入导出设计

日期：2026-07-17
状态：已确认设计

## 1. 背景与目标

客户资料是公司核心数字资产。当前极享OS已具备单客户的查看、编辑、分配、放弃到公海、公海领取、跟进、待办和订单关联能力，但缺少统一的批量管理、日常客户导入导出和重复客户合并能力。

本项目的目标是：

1. 建立可配置的客户功能权限树，不在业务代码中写死角色名称。
2. 建立可复用的客户批量任务引擎，支持跨页选择、当前筛选结果全选、预检、异步执行、失败报告和审计。
3. 使用一套极享OS标准客户模板完成日常导入，不为每个外部CRM单独开发导入器。
4. 让客户导出严格受功能权限、客户数据范围和敏感字段权限控制。
5. 建立重复客户治理流程，安全迁移关联业务，保留来源、归因和操作轨迹。

## 2. 设计原则

- **权限可配置**：角色只是权限和数据范围的载体，业务服务不判断“销售”、“主管”或“超级管理员”等角色名称。
- **范围不扩张**：批量操作和导入导出必须复用 `customers` 数据范围，不因为是批量入口而扩大可见或可操作数据。
- **服务端最终裁决**：前端只用于展示和收集意图；服务端在预检和正式执行时均重新校验权限、数据范围、客户状态和数据版本。
- **先预检，后写入**：批量修改、导入、导出和合并均先展示影响范围和阻断原因，用户确认后才创建任务。
- **不静默丢数据**：所有部分失败均生成明细和可下载报告；不用“导入完成”隐藏跳过、重复或无权数据。
- **稳定身份关联**：客户负责人、操作人和关联客户优先使用稳定ID，姓名只用于展示和历史快照。
- **可追溯和可恢复**：记录批次、操作人、原因、输入范围、客户清单、前后状态和执行结果；删除使用软删除，合并保留副客户和合并账本。

## 3. 功能范围

### 3.1 批量操作

- 批量转让／分配客户
- 批量放弃客户到公海
- 批量设置客户进展
- 批量添加或移除客户标签
- 批量添加客户待办
- 批量软删除客户
- 批量导出客户

### 3.2 数据治理

- 极享OS标准客户模板下载
- 客户文件预检、导入和失败报告
- 按权限和字段范围导出
- 重复客户候选组、人工合并和有条件撤销
- 批量任务与审计记录

### 3.3 不在范围内

- 不开发按CRM厂商区分的导入器；外部数据由数据提供方转换为极享OS标准模板。
- 不实现共享客户和撤销客户共享。
- 不实现批量发短信、批量添加待拨打列表或外呼系统集成。
- 不引入独立“客户分组”模型；客户分类继续使用可配置标签。
- 不自动合并全部重复客户。
- 导入发现重复时不自动覆盖或更新已有客户。

## 4. 权限树与数据范围

### 4.1 权限树

```text
客户管理
├─ 客户访问
│  ├─ 查看客户列表
│  └─ 查看客户资料
│
├─ 客户资料维护
│  ├─ 新建客户
│  ├─ 编辑客户资料
│  ├─ 设置客户进展
│  ├─ 设置客户标签
│  ├─ 设置客户待办
│  ├─ 修改客户归因
│  └─ 删除客户
│
├─ 客户归属与公海
│  ├─ 转让／分配客户
│  ├─ 放弃客户到公海
│  └─ 领取公海客户
│
├─ 批量与数据治理
│  ├─ 批量管理客户
│  ├─ 导入客户
│  ├─ 导入客户归因覆盖
│  ├─ 导出客户
│  ├─ 导出客户敏感字段
│  ├─ 合并重复客户
│  ├─ 撤销客户合并
│  ├─ 取消批量任务
│  └─ 查看批量任务及审计记录
│
├─ 客户业务
│  ├─ 新建客户订单
│  └─ 查看客户订单
│
└─ 客户智能能力
   ├─ 查看客户画像
   └─ 使用AI客户名片
```

树上的中间节点仅用于分组和全选，服务端只识别叶子权限。勾选父节点等于勾选当前所有子节点，部分勾选显示半选状态。

### 4.2 原子权限组合

批量操作不为每个动作复制一套权限，而是组合判断：

| 批量动作 | 必须同时拥有 |
| --- | --- |
| 批量转让 | 批量管理客户 + 转让／分配客户 |
| 批量放弃 | 批量管理客户 + 放弃客户到公海 |
| 批量设置进展 | 批量管理客户 + 设置客户进展 |
| 批量设置标签 | 批量管理客户 + 设置客户标签 |
| 批量添加待办 | 批量管理客户 + 设置客户待办 |
| 批量删除 | 批量管理客户 + 删除客户 |
| 批量导出 | 批量管理客户 + 导出客户 |

导入、合并和撤销合并不需要“批量管理客户”。导入必须同时拥有“导入客户”和“新建客户”，如果归属其他员工或公海则再组合校验对应归属权限。合并和撤销合并使用各自独立权限。导出姓名、公司、进展等普通字段需要“批量管理客户”和“导出客户”；导出手机号、微信、邮箱等字段还必须拥有“导出客户敏感字段”。

### 4.3 数据范围

客户域使用独立 `customers` 数据范围。目标档位为：

```text
仅本人
仅本部门
本部门及所有下级部门
全部客户
```

导入时，用户只能把客户归属给其可管理范围内的在职员工。合并时，操作者必须对主客户和全部副客户均有数据操作权限。导出时，服务端根据导出时的实时数据范围重新生成清单。

数据范围只决定“哪些客户可见或可操作”，不代替功能权限。例如拥有“全部客户”范围但没有“转让／分配客户”的用户，仍然不能转让客户。

服务端明确区分：

- `canReadCustomer`：沿用现有可见规则，可因客户负责人、线索贡献人或公海身份获得查看权。
- `canManageCustomer`：只有客户归属在当前用户的 `customers` 可操作范围内时才成立；仅因贡献人或公海可见不授予编辑、转让、放弃、删除、设标签或合并权。公海领取只走“领取公海客户”的独立校验。

所有单客户与批量写命令在预检和正式执行时均调用同一 `canManageCustomer`。导出可使用 `canReadCustomer`，但仍受导出和敏感字段权限限制。

`canManageCustomer` 只使用稳定 `ownerId` 计算，不用姓名作写权限依据。第一批上线前复用现有负责人身份回填服务：唯一匹配时补齐ID，同名、已离职或无法匹配时标记为待修复。负责人ID未解析的客户仍可按旧规则只读查看，但除专用身份修复操作外的写命令一律失败关闭。

### 4.4 兼容迁移

- 当前“分配客户”同时覆盖转让和放弃。上线时将拥有原分配权限的角色一次性迁移为“转让／分配客户”与“放弃客户到公海”，保持现有行为。
- 当前“编辑客户”的单笔更新已包含基础资料、标签和归因字段，待办的创建、编辑、重开和取消也使用该权限。因此旧权限迁移为“编辑客户资料”、“设置客户标签”、“设置客户待办”和“修改客户归因”，以保留已有有效行为；但不迁移为“设置客户进展”，因为现有通用编辑白名单不允许修改生命周期。
- 当前删除能力的角色名称判断在迁移后移除。迁移脚本仅向上线前确实拥有删除有效权的角色ID授予“删除客户”，业务代码不再识别“超级管理员”等名称。
- 现有 `self` 和 `all` 保持不变；现有 `department` 自动包含下级部门，迁移为新的“本部门及所有下级部门”枚举，保证上线前后可见范围不收窄。“仅本部门”作为新选项供后续配置。
- 迁移后所有叶子权限均可由管理员独立调整，后续不再自动扩权。
- 内置角色可提供默认权限模板，但业务代码不识别内置角色名称。
- 新旧权限并行的过渡版中，所有旧单笔路由和新批量路由都调用新叶子权限策略；不允许只限制批量入口而保留可绕过的旧写接口。
- 通用客户更新接口按字段分组校验叶子权限：基础资料、进展、标签和归因不能仅靠一个“编辑客户”全部通过。一个请求同时修改多组字段时，必须同时拥有全部对应叶子权限，否则整个请求拒绝，不部分写入。
- 权限、数据范围和负责人ID迁移均使用显式版本号和幂等记录，可安全重跑，不重复授权、不覆盖上线后管理员已调整的配置。

## 5. 页面结构

### 5.1 客户管理入口

客户管理保留现有“客户列表”和“公海池”，并根据权限增加：

- **重复客户治理**：展示疑似重复组、合并状态和历史合并。
- **批量任务**：展示本人有权查看的导入、导出、批量修改和合并任务。

“导入客户”和“导出客户”位于客户列表顶部，按权限显示。不在系统设置中建立日常导入入口。

### 5.2 批量选择

- 表格第一列增加复选框，表头支持本页全选和半选状态。
- 选中数据后显示固定批量操作条，仅显示当前用户可执行的动作。
- 用户可以选择“当前页”或“当前筛选结果的全部客户”。
- “全部客户”仅代表服务端按当前筛选和客户数据范围计算的结果，不代表绕过数据范围的整库全选。
- 任务创建时冻结客户ID清单和预检版本，不在执行途中重新扩大范围。

## 6. 批量处理流程

```text
勾选客户
→ 选择当前页／当前筛选结果全部客户
→ 选择批量动作
→ 填写动作参数和原因
→ 服务端预检
→ 展示可执行、不可执行和原因
→ 用户确认
→ 创建批量任务
→ 展示进度和结果
→ 下载失败或完整结果报告
```

### 6.1 数量规则

- 单次任务最多处理 10,000 个客户；超过时要求缩小筛选范围。
- 200 个及以下可在创建任务后立即执行并在当前页等待结果。
- 超过 200 个或使用“当前筛选结果全选”时，统一进入后台任务。
- 即使 200 个及以下，也必须先将任务和明细持久化后再执行；“当前页等待”只是交互差异，不是两套执行逻辑。
- 页面离开不会取消任务，任务中心持续显示进度。
- 任务创建人可查看自己的任务摘要；拥有“查看批量任务及审计记录”的用户，可查看其当前数据范围内客户所对应的任务和明细。明细查询始终按当前权限和数据范围逐项过滤；当任务同时包含范围内外数据时，不向查看者暴露范围外明细和字段。无任务可见权限不得下载结果文件。

### 6.2 执行语义

- 预检生成限时 10 分钟且只能成功消费一次的确认令牌。令牌绑定操作人、操作类型、冻结客户ID清单、参数哈希和预期数据版本；导入令牌还绑定原文件哈希、模板版本和规范化行清单哈希。正式执行通过原子状态转移消费令牌，不得换参、换文件或重放。
- 每个任务使用唯一幂等键，重复点击或网络重试不会重复执行。
- 为避免大任务长时间锁表，按固定客户ID顺序分批执行，每批使用数据库事务和行锁。
- 正式执行时如客户已变更、权限已收回、负责人已变化或状态不允许，该项标记为失败，不用预检时的旧结果强行写入。
- 批量任务允许部分成功，但必须清晰显示成功、失败和未处理数量及逐行原因。
- 合并任务是例外：同一重复组的全部迁移必须在一个事务内全部成功，任意一步失败即整组回滚。
- 后台工作者使用数据库任务状态作为事实来源。任务通过租约拥有者、租约代次、心跳和条件更新原子抢占；每批提交前重新校验租约代次和取消状态，旧工作者在租约失效后不得继续提交。服务重启后自动恢复排队任务，并重新抢占超过心跳时限的执行中任务。
- 任务明细和其业务副作用使用唯一幂等键，已成功明细不再重复执行。只有网络中断、工作者崩溃和可重试数据库错误自动重试；权限、数据范围、参数、版本和业务状态冲突均为不可重试失败。
- 任务创建人可取消自己的任务。其他用户必须拥有“取消批量任务”，且任务的全部客户都在其当前数据范围内。“查看批量任务及审计记录”只是读权限，不授予取消能力。排队中任务可直接取消；执行中任务只停止尚未开始的分批，已提交事务不回滚，结果明确区分成功、失败和已取消未处理数。
- 任务状态机固定为 `queued → running → succeeded | partial_failed | failed`；取消路径为 `queued | running → cancel_requested → cancelled`。系统只能按状态机条件更新，不得直接覆盖终态。合并任务在进入单事务后不接受取消，只能等待整体提交或回滚。

### 6.3 动作规则

| 动作 | 需要的输入 | 主要预检 | 执行结果 |
| --- | --- | --- | --- |
| 转让／分配 | 目标员工ID、必填原因 | 目标员工在职、可承接客户、在可管理范围 | 更新客户和关联线索归属；尚未完成的客户待办转给新负责人，保留历史执行人 |
| 放弃到公海 | 必填原因 | 客户非公海、操作人有权放弃 | 释放归属并同步关联线索；未完成待办标记为已取消并记录原因，不删除历史 |
| 设置进展 | 目标生命周期状态、必填原因 | 目标状态启用、符合生命周期转移表 | 更新进展及变更时间，写入客户动态与审计 |
| 设置标签 | 添加或移除、标签ID | 标签启用、分组选择模式无冲突、不超过上限 | 仅添加或移除指定标签，不默认替换客户全部标签 |
| 添加待办 | 标题、内容、到期时间、执行方式 | 客户有有效负责人，待办数据有效 | 每个客户创建一个指派给其当前负责人的待办；幂等键防止重复创建 |
| 删除 | 必填原因、高风险二次确认 | 无订单、交付、售后、财务等关联业务，未处于已合并状态 | 软删除并进入回收站，不物理删除客户资料 |

客户生命周期配置增加“可转入状态”转移表。迁移时根据现有启用状态生成与当前行为等价的默认转移表，后续由管理员配置；单客户和批量设置进展共用同一服务端转移校验。

公海、已成交等由归属或关联业务命令驱动的系统状态，不出现在手工“设置客户进展”目标中。例如进入公海必须调用放弃命令，不能用进展权限绕过归属校验。

转让时重派未完成待办、放弃时取消未完成待办，以及删除时全关联阻断等规则，同时下沉到单客户命令服务。批量任务仅编排这些单项原子能力，不维护另一套业务规则。

“设置客户待办”控制创建、编辑、重开、取消和代他人完成待办。待办实际执行人可完成自己的待办，不需要该管理权限，但必须仍可查看对应客户。

## 7. 极享OS标准客户导入

### 7.1 模板原则

- 只维护一套极享OS标准客户模板，文件和工作表内标记模板版本，首版为 `JixiangOS-Customer-Import-V1`。
- 系统保持对已发布模板版本的向后兼容；新增字段时优先使用可选列，不直接使旧模板失效。
- 外部CRM的数据必须先转换为该模板；极享OS不识别或分支处理外部CRM品牌。
- 模板动态带出当前可用客户进展、客户等级、标签、线索来源和可分配员工的填写说明或下拉数据。

### 7.2 导入字段

客户姓名必填，手机号和微信至少填写一项。标准模板支持：

- 客户姓名、手机号、微信号、邮箱、公司名称
- 负责人账号／稳定员工ID、客户进展、客户等级
- 客户标签、线索来源、资源归属、行业、城市
- 线索录入人、线索贡献人、原销转人员
- 备注

负责人只能使用稳定员工ID或全局唯一账号解析。负责人姓名可在模板中作为便于阅读的辅助列；如同时填写的ID、账号和辅助姓名不指向同一人，整行阻断，不猜测匹配。

线索录入人默认固定为导入操作人，线索贡献人和原销转人默认留空。只有同时拥有“导入客户归因覆盖”的用户才可导入这三类归因字段；没有该权限而行内存在覆盖值时，预检阻断并明确报错，不静默忽略。所有归因覆盖记录原值、目标员工ID和操作人。

### 7.3 导入流程

```text
下载标准模板
→ 选择文件
→ 文件与模板版本校验
→ 逐行字段校验和标准化
→ 文件内查重
→ 客户库与线索库查重
→ 归属、数据范围和配置项校验
→ 展示预检结果
→ 确认导入可导入行
→ 后台任务写入
→ 下载完整结果报告
```

首版只支持 `.xlsx`，文件最大 20 MB，单文件最多 10,000 条数据。空行不计入数据量，但不允许擅自修改必要表头或模板版本。

文件解析在隔离任务中执行，同时限制压缩包展开体积、工作表数量、单元格数量和解析时间。导入值一律按纯文本和允许的日期／数字类型读取，不执行公式、宏、脚本、外部链接或嵌入对象。导出时对以 `=`、`+`、`-` 或 `@` 开头的文本做公式注入防护。

导入向导必须选择默认归属：归属当前用户、归属指定在职员工，或直接进入公海。模板行内有合法负责人ID或唯一账号时覆盖默认归属，空白时使用向导默认值。目标员工必须在职、可承接客户且位于操作人可管理范围。

导入权限不自动扩大客户归属能力：

- 归属当前用户：需要“导入客户”与“新建客户”。
- 归属其他员工：还需要“转让／分配客户”，且目标员工在可管理范围内。
- 进入公海：还需要“放弃客户到公海”。

任一行覆盖默认归属时，服务端按该行实际归属重新组合校验权限与数据范围。

### 7.4 重复处理

- 手机号和微信号经标准化后参与唯一身份校验。
- 文件内重复行和与现有客户／线索重复的行都归入“重复数据”。
- 导入时只跳过重复行并生成重复报告，不自动覆盖、更新或合并现有客户。
- 重复报告可一键创建重复客户治理候选组，但合并仍需人工确认。
- 查重在公司全局执行以阻止跨数据范围的新重复，但不因此泄露范围外客户。操作人对冲突客户或线索没有查看权时，报告只显示“系统中已存在相同联系方式”，不返回对象ID、姓名、负责人或所属部门。候选组可在后台生成，但只对具备合并权限且能管理组内全部客户的用户展示。

### 7.5 预检结果

每行只能落入一个最主要结果类型，同时可携带多个错误说明：

- 可导入
- 字段错误
- 重复客户／线索
- 负责人不存在、已离职或姓名不唯一
- 无权分配
- 标签、来源、等级或进展不存在
- 系统执行失败

下载报告保留原始行号、原数据和明确原因，便于修改后再次上传。

## 8. 客户导出

### 8.1 导出范围

- 导出已勾选客户。
- 导出当前筛选结果中有权查看的全部客户。
- 导出时服务端重新应用实时客户数据范围；预检后权限被收回的客户不会进入文件，并记录为未导出。

### 8.2 字段和敏感数据

- 用户可在导出对话框中选择可用字段，不可见字段不出现在选项中。
- 手机号、微信号和邮箱属于敏感字段。
- 同时拥有“批量管理客户”和“导出客户”，但没有“导出客户敏感字段”时，导出文件中的敏感值按统一规则脱敏，服务端不返回完整值。
- 导出文件的说明页记录导出人、导出时间、批次号、数据范围、筛选摘要和是否包含敏感字段。

### 8.3 任务和下载

- 导出一律建立审计任务；小文件可在当前页等待，大文件在后台生成。
- 单次导出最多 10,000 条，超过时要求缩小筛选范围。
- 下载链接必须经身份校验且 24 小时后过期，不使用可公开访问的静态URL。
- 每次下载前重新校验导出权限、敏感字段权限和当前数据范围。如任务生成后任一权限或范围收紧，原文件不得下载，用户需按当前权限重新创建导出任务。
- 任务记录生成数、脱敏数、跳过数、失败原因、下载次数和最后下载时间。

导入原文件、导入结果报告和导出文件都使用受控文件存储，静态加密，不写入公开目录。文件默认保留 30 天，管理员可配置更短期限；过期后删除文件并吊销下载，任务摘要和追加式审计记录继续保留。导入结果报告与导出文件使用同级别的身份、权限和当前数据范围复核。

任务创建人下载原始或结果文件时，仍需拥有对应的导入／导出权限与当前数据范围。其他审计用户仅有“查看批量任务及审计记录”时只能查看脱敏摘要；下载原始文件还必须拥有对应业务权限，且任务全部客户在其当前数据范围内。只能看到混合任务部分明细时不允许下载原文件，如需数据应按当前权限新建导出。

## 9. 重复客户治理

### 9.1 候选规则

系统只生成疑似重复候选组，不自动合并：

- 标准化手机号相同：高可信重复。
- 标准化微信号相同：高可信重复。
- 客户姓名与公司名称等组合相同：疑似重复，必须人工判断。
- 导入重复报告或用户手动选择：手动候选组。

系统不使用模糊姓名相似度直接判定重复，避免同名客户被误合并。

### 9.2 合并向导

```text
选择 2–10 个疑似重复客户
→ 指定主客户
→ 逐字段对比并选择保留值
→ 查看关联业务迁移清单
→ 处理归属、标签和归因冲突
→ 填写合并原因
→ 最终确认
→ 单事务执行
→ 生成合并账本
```

### 9.3 字段规则

- 主客户ID和客户编号永久保留。
- 主客户空白字段可默认使用副客户非空值。
- 姓名、手机、微信、公司、负责人和进展冲突必须明确选择，不静默覆盖。
- 标签默认取并集；同一单选标签组冲突时必须人工选择。
- 负责人不同时必须明确选择合并后负责人。
- 来源、资源归属、线索录入人、线索贡献人和原销转人员保留原始快照和来源历史，不用单一主客户值覆盖归因证据。

### 9.4 关联数据

合并需要显式盘点并迁移：

- 订单和订单申请
- 交付、售后、退款和财务关联记录
- 线索及其跟进记录
- 客户跟进记录、附件和动态
- 客户待办
- 标签、成长记录和AI画像

关联业务的原始名称、金额、归因和审核快照不因客户合并而改写；只更新后续定位所需的稳定客户ID关联。

上线合并前必须完成一次关联盘点和数据回填：订单、交付、售后、财务、线索、跟进、待办和附件等关联域必须使用稳定 `customerId`。仅有客户姓名且无法唯一回填的历史记录进入数据修复清单，相关客户组在修复完成前不允许合并。

合并服务使用显式“客户关联迁移注册表”，禁止在合并代码中临时全库猜测。首版注册表为：

| 关联域 | 查询和写入路径 | 合并与回滚 |
| --- | --- | --- |
| `LeadRecord` | `data.customerId` 与线索业务快照 | 改为主客户ID；账本记录每条原ID和行版本 |
| `BusinessRecord: aaos_orders` | 顶层 `customerId` 与 `data.customerId` | 两处同步写主客户ID；不改历史客户姓名、金额和销售快照 |
| `BusinessRecord: aaos_order_applications` | 顶层 `customerId`、`data.customerId` 与 `data.orderData.customerId` | 所有ID路径一致迁移；审核提交快照保留 |
| `BusinessRecord: aaos_deliveries` | 顶层 `customerId` 与 `data.customerId` | 迁移定位ID；交付快照不改写 |
| `BusinessRecord: aaos_refunds` | 顶层 `customerId` 与 `data.customerId` | 迁移定位ID；退款审核和金额快照保留 |
| `BusinessRecord: aaos_recovery_orders` | 顶层 `customerId` 与 `data.customerId` | 迁移定位ID；挽回客户和分账快照保留 |
| `BusinessRecord: aaos_service_tickets` | 顶层 `customerId` 与 `data.customerId` | 迁移定位ID；工单历史保留 |
| `BusinessRecord: aaos_opportunities` | 顶层 `customerId` 与 `data.customerId` | 迁移定位ID；商机历史保留 |
| `BusinessRecord: aaos_commissions / aaos_finance` | 顶层或JSON内的 `customerId`，主要业务关联仍以 `orderId` 为准 | 仅迁移客户定位ID；不重算、不改归因和金额 |
| `CustomerTodo` | 列 `customerId`、`customerName` | 迁移ID，名称改为主客户当前展示名；账本保留原值 |
| 客户JSON子资料 | 跟进、动态、成长记录、标签和附件引用 | 按记录ID去重后迁入主客户，附件只保留受控引用和哈希，不在账本复制文件内容 |
| `BusinessRecord: aaos_ai_cards` | `data.subjectType=customer` 且 `data.subjectId` | 副客户画像保留为历史来源，主客户画像标记待重新生成，不盲目拼接AI结论 |

每个注册项实现查询、加锁、迁移、版本快照、冲突检查和撤销六个操作，并由集成测试验证。新增含客户关联的业务域时，未注册前构建检查和合并预检必须失败关闭，不允许带未知关联执行合并。

### 9.5 副客户和旧链接

副客户不物理删除，而是保留：

- `mergedIntoId`
- `mergedAt`
- `mergedById` 和 `mergedByName`
- `mergeBatchId`
- 合并前快照

从旧客户ID或旧链接访问时，系统返回“已合并”状态和主客户ID，前端提示后跳转到主客户。副客户不出现在普通客户列表中。所有客户读取和写命令首先解析 `mergedIntoId`：读取返回可跳转结果，写命令对副客户返回 409 并提供主客户ID，不允许通过旧单笔接口继续修改副客户。

### 9.6 有条件撤销

- 合并后 72 小时内、且主客户和副客户未产生新业务变化时，拥有撤销合并权限的用户可自动撤销。
- 撤销使用合并账本和合并前快照恢复副客户、字段和原关联。
- 如已产生新订单、新跟进、新待办、新附件或再次合并，系统阻止自动撤销并给出冲突清单。
- 超过 72 小时后不提供自动撤销，仅可通过有记录的数据治理流程人工处理。

“未产生新业务变化”由合并账本保存的行版本和事件水位、客户审计事件，以及全部关联域的 `createdAt` / `updatedAt` 共同判定，不依赖前端状态。合并和撤销都按“主客户ID → 副客户ID升序 → 关联域固定顺序”加锁，撤销前重算冲突。客户、关联记录、联系方式身份、合并账本和审计事件必须在同一事务中全部提交或整体回滚。如因冲突而整体回滚，失败审计在回滚后使用独立事务追加。

## 10. 数据模型

### 10.1 客户批量任务

`CustomerBatchPrecheck`：

- `id`、`actorId`、`operation`、`tokenHash`、`status`
- `selectionHash`、`inputHash`、`fileHash`、`normalizedRowsHash`
- `customerVersionManifest`、`expiresAt`、`consumedAt`

`CustomerBatchJob`：

- `id`、`operation`、`status`
- `selectionMode`（`ids` 或 `filter_snapshot`）
- `filterSnapshot`、`selectedCustomerIds`、`frozenCustomerCount`
- `input`、`reason`、`idempotencyKey`
- `actorId`、`actorName`、`actorDepartmentId`
- `totalCount`、`successCount`、`failedCount`、`skippedCount`、`cancelledCount`
- `resultFileId`、`expiresAt`
- `leaseOwner`、`leaseEpoch`、`leaseExpiresAt`、`heartbeatAt`、`cursor`
- `attemptCount`、`lastError`、`retryOfJobId`
- `createdAt`、`startedAt`、`finishedAt`、`cancelRequestedAt`、`cancelledAt`

`CustomerBatchJobItem`：

- `jobId`、`customerId`、`status`、`errorCode`、`errorMessage`
- `expectedUpdatedAt`、`beforeHash`、`afterHash`
- `beforeSnapshot`、`afterSnapshot`
- `sourceRowNumber`、`rowHash`、`normalizedPayloadHash`、`resultCustomerId`
- `idempotencyKey`、`attemptCount`、`retryable`
- `startedAt`、`finishedAt`

预检、任务和任务明细使用独立数据表，不只存入前端 `localStorage` 或客户JSON内部。数据库约束 `(actorId, operation, idempotencyKey)` 唯一，同键不同参数返回 409；客户动作明细使用可空 `sourceRowNumber` 与唯一约束 `(jobId, customerId)`，导入明细使用可空 `customerId` 与唯一约束 `(jobId, sourceRowNumber)`。明细幂等键另设唯一约束，不依赖应用层先查后写。导入明细写入成功后再记录 `resultCustomerId`。

### 10.2 审计事件

`CustomerAuditEvent` 为追加式记录：

- 客户ID、批次ID、操作类型
- 操作人ID与姓名快照
- 原因、参数摘要、前后快照或哈希
- 成功或失败结果
- 请求ID、幂等键、IP和时间

客户详情动态可以展示该审计事件的业务友好摘要，但不代替独立审计记录。

审计快照使用字段白名单，不复制附件内容。手机号、微信、邮箱等敏感值在普通审计视图中脱敏，如为恢复所必须保留完整值则进行字段级加密，密钥与数据库分离管理。

### 10.3 联系方式唯一身份

联系方式不由每条线索和客户各自争抢唯一值，而是使用“唯一身份 + 多业务关联”模型：

`ContactIdentity`：

- `id`、`type`（手机号或微信）、`normalizedHash`、`hashKeyVersion`、`status`
- `encryptedNormalizedValue`、`canonicalCustomerId`、`conflictReason`、`createdAt`、`updatedAt`
- `normalizedHash` 使用服务端密钥生成的稳定HMAC，数据库约束 `(type, normalizedHash)` 唯一；不以明文联系方式作索引键

`ContactIdentityLink`：

- `identityId`、`entityType`（线索或客户）、`entityId`
- `linkStatus`、`source`、`createdAt`、`endedAt`
- 数据库约束 `(identityId, entityType, entityId)` 唯一

线索转客户时，原线索关联保留，同一 `ContactIdentity` 新增客户关联并将 `canonicalCustomerId` 指向该客户，因此正常转化不会被误判为线索与客户冲突。

第一批上线前执行全量回填：

- 仅有一个有效客户时，将其设为 `canonicalCustomerId`。
- 历史上已有多个有效客户共用联系方式时，保留全部关联，将身份标记为 `conflict` 且暂不设置主客户，同时生成重复候选组。在第三批人工合并前不破坏历史数据，但所有新建、编辑、转客户和导入都不得再扩大该冲突。

新建、编辑、线索转客户、导入、合并和撤销合并都在同一事务内更新身份和关联表。合并时将副客户关联结束并把主客户设为唯一主客户；撤销时按合并账本恢复，如联系方式已被新的有效客户占用则阻止自动撤销。

### 10.4 重复候选组

`CustomerDuplicateGroup` 记录：

- `id`、`rule`、`confidence`、`status`
- `customerIds`、`contactIdentityId`、`sourceJobId`
- `createdById`、`createdAt`、`resolvedAt`、`mergeLedgerId`

联系方式回填、导入重复报告和用户手动选择都写入同一候选组模型。第一批只需要生成和审计冲突候选，第三批再上线完整治理界面和合并流程。

### 10.5 合并账本

`CustomerMergeLedger` 记录：

- 主客户ID、副客户ID列表、候选规则
- 字段裁决结果和全部合并前快照
- 迁移的关联记录ID、原关联和新关联
- 合并原因、操作人、合并时间和撤销截止时间
- 撤销状态、撤销人和撤销失败冲突

## 11. 服务边界与接口

实现拆分为独立深模块，避免继续扩张现有客户列表页和客户命令服务：

- `customerBatchService`：预检、冻结清单、任务调度、逐项执行和结果。
- `customerImportService`：模板、文件解析、标准化、查重、预检和写入。
- `customerExportService`：字段权限、脱敏、生成文件、限时下载和审计。
- `customerMergeService`：候选组、字段裁决、关联迁移、合并账本和撤销。
- `customerAuditService`：统一追加审计事件。

实现接口：

```text

GET  /api/customer-batch-jobs
GET  /api/customer-batch-jobs/:id
GET  /api/customer-batch-jobs/:id/items
GET  /api/customer-batch-jobs/:id/result-file
POST /api/customer-batch-jobs/:id/cancel

GET  /api/customer-import/template
POST /api/customer-import/precheck
POST /api/customer-import/jobs

POST /api/customer-export/jobs
```

## 12. 错误与边界处理

- **数据已变更**：预检后 `updatedAt` 发生变化时该项执行失败，返回“客户已变更，请重新预检”。
- **权限或范围改变**：执行时以实时权限为准，不允许用旧预检结果继续写入。
- **目标员工离职或停用**：转让和导入失败，不回退到同名员工。
- **标签或进展在任务中被停用**：相关项失败并记录原因。
- **任务重试**：任务创建人可将可重试失败项带入新的标准预检流程，并必须仍拥有原操作功能权限和数据范围。重试创建新任务，不重新执行已成功或不可重试项，并保留 `retryOfJobId`。审计读权限不授予重试能力。
- **导入超时或断线**：任务继续在服务端执行，前端恢复后根据任务ID继续轮询。
- **文件解析失败**：不创建写入任务，返回模板版本、工作表或表头错误。
- **合并循环**：禁止将主客户合并到其历史副客户，禁止已合并客户再作为主客户。
- **联系方式冲突**：由唯一身份约束阻止并发写入，转换为可读的重复冲突结果。
- **客户已删除或已合并**：批量修改不再操作，结果明细指向回收站或主客户。

## 13. 验收标准

1. **Given** 某角色有“批量管理客户”但没有“转让／分配客户”，**When** 勾选多个客户，**Then** 不显示批量转让，直接请求接口返回 403。
2. **Given** 用户客户数据范围为仅本部门，**When** 选择当前筛选结果全选，**Then** 预检和执行都只包含本部门可见客户，不包含下级部门。
3. **Given** 批量任务预检完成，**When** 其中一个客户在执行前变更负责人，**Then** 该客户失败并显示“数据已变更”，其他合法客户继续执行。
4. **Given** 任务已成功处理部分客户，**When** 用户重复提交相同幂等键，**Then** 系统返回原任务，不再生成重复动态或待办。
5. **Given** 导入文件包含 1 条客户库重复和 1 条线索库重复，**When** 执行预检，**Then** 两行均被标记重复并不可导入，报告说明重复类型，且只在操作人有查看权时才返回冲突对象资料。
6. **Given** 外部CRM数据已转换为合法的极享OS V1模板，**When** 上传文件，**Then** 导入程序不关心数据原CRM品牌，按统一规则预检。
7. **Given** 导入文件的负责人不在操作者可管理范围，**When** 预检，**Then** 该行显示“无权分配”且不写入客户。
8. **Given** 用户有批量管理和导出客户权限，但没有导出敏感字段权限，**When** 导出包含手机号和微信字段，**Then** 生成文件中该字段已脱敏，服务端日志和网络响应不暴露完整值。
9. **Given** 用户同时有批量管理、导出客户和敏感字段权限，**When** 选择敏感字段导出，**Then** 文件包含完整值且审计记录明确标记“包含敏感字段”。
10. **Given** 两个客户负责人不同，**When** 用户未选择合并后负责人，**Then** 合并预检不通过。
11. **Given** 副客户包含订单、线索、跟进和待办，**When** 合并成功，**Then** 关联ID迁移至主客户，历史金额和归因快照不被改写，副客户旧链接跳转主客户。
12. **Given** 合并迁移的任何一步失败，**When** 事务结束，**Then** 主客户、副客户和全部关联数据保持合并前状态。
13. **Given** 合并在 72 小时内且没有任何后续变更，**When** 有撤销权限的用户确认撤销，**Then** 按合并账本恢复副客户和原关联，并生成撤销审计记录。
14. **Given** 合并后主客户产生了新订单，**When** 申请撤销，**Then** 系统阻止自动撤销并列出冲突。
15. **Given** 客户存在业务关联，**When** 批量删除预检，**Then** 该客户被阻止且其他合法客户可继续软删除。
16. **Given** 用户离开批量任务页面，**When** 后台任务完成，**Then** 用户再次进入可查看完整结果和下载报告。
17. **Given** 管理员修改某角色的客户叶子权限和数据范围，**When** 该角色用户重新进入客户页，**Then** 页面入口、批量动作、服务端命令和可处理数据均按新配置生效，不需要修改代码。
18. **Given** 用户只有导入与新建客户权限，**When** 导入行指定其他员工或公海为归属，**Then** 预检依据缺少分配或放弃权限阻止该行。
19. **Given** 后台任务执行中服务重启，**When** 工作者恢复，**Then** 任务从已提交明细之后继续，不重复处理已成功客户。
20. **Given** 历史关联记录只有客户姓名且无法唯一回填，**When** 尝试合并该客户组，**Then** 预检阻止合并并返回待修复的关联记录清单。
21. **Given** 用户仅因线索贡献人身份可查看某客户，**When** 调用单笔或批量编辑、转让或删除，**Then** `canManageCustomer` 阻止写入，但合法查看不受影响。
22. **Given** 用户有任务审计读权限但没有取消权限，**When** 尝试取消他人任务，**Then** 界面不显示操作且接口返回 403。
23. **Given** 导入者没有“导入客户归因覆盖”，**When** 模板填写了其他线索录入人或贡献人，**Then** 预检阻止该行而不静默修改归因。
24. **Given** 一条线索转为客户，**When** 线索和新客户共用同一手机号，**Then** 两者关联同一 `ContactIdentity`，客户成为主客户，转化不被当成跨客户重复。
25. **Given** 已生成的导出任务后来被收回敏感字段权限或缩小数据范围，**When** 用户下载旧文件，**Then** 系统拒绝下载并要求按当前权限重新导出。
26. **Given** 导入行与操作人数据范围外的客户联系方式重复，**When** 预检，**Then** 该行被阻止，但报告不泄露冲突客户的ID、姓名、负责人或部门。
27. **Given** 用户有设置客户进展权限但没有放弃到公海权限，**When** 尝试把进展设为公海，**Then** 系统状态不在可选项中且服务端拒绝请求。

## 14. 指标与运营观测

上线后记录但不默认公开展示以下指标：

- 每类批量任务的数量、成功率、失败率和平均耗时
- 导入数量、重复率、无效行率和重试率
- 导出数量、含敏感字段导出数和下载次数
- 疑似重复组数、合并数、撤销数和被阻止撤销数
- 批量操作中因权限、数据变更或状态冲突被阻止的数量

## 15. 分阶段实施

功能范围不缩减，按依赖关系分三个可验收批次交付：

### 第一批：权限与批量底座

- 客户权限树拆分、旧角色兼容迁移、新部门范围枚举和 `canReadCustomer` / `canManageCustomer` 统一策略
- 批量选择、一次性预检令牌、任务状态机、租约工作者、任务明细和审计底座
- 批量转让、放弃、进展、标签、待办和软删除
- 客户关联迁移注册表、历史 `customerId` 回填清单和删除全关联阻断
- 联系方式身份与多业务关联、历史冲突回填、重复候选记录和新写入冲突阻断

### 第二批：标准导入导出

- 标准模板V1、动态字典和版本兼容
- 隔离文件解析、导入归属与归因校验、查重、导入任务和失败报告
- 导出字段选择、敏感字段脱敏、文件生成、受控存储、过期删除和下载时重新授权

### 第三批：重复客户治理

- 重复候选组治理界面和手动候选组
- 字段对比、主客户裁决和显式关联注册表预检
- 事务合并、副客户跳转、合并账本和 72 小时有条件撤销

## 16. 发布前校验

- 每一批都必须覆盖权限绕过、数据范围越权、并发冲突、重试幂等、部分失败和大数据量测试。
- 权限和数据范围迁移必须在生产数据副本上生成上线前后有效权限差异报告，任何非预期扩权或缩权都阻止发布。
- 批量工作者必须验证服务重启、租约过期接管、双工作者争抢、执行中取消和不可重试错误不重试。
- 联系方式回填必须输出唯一主客户数、历史冲突数、无效联系方式数和候选组数，且支持幂等重跑与数据校验。
- 导入必须用旧模板、新模板、非法模板、重复文件、超大文件和中断恢复场景验收。
- 导出必须检查脱敏文件、完整敏感文件、过期链接、越权下载和审计记录。
- 合并必须使用真实关联订单、线索、跟进、待办和附件的复杂样本验证全成功与中途回滚。
- 必须通过类型检查、相关模块测试、完整测试套件、生产构建和真实浏览器验收后才能发布。
