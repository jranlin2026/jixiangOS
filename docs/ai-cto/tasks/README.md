# AI CTO任务目录

## 文件命名

```text
TASK-YYYYMMDD-序号-简短任务名.md
```

示例：`TASK-20260712-001-客户自动跟进.md`。

## 编号规则

- 同一天从 `001` 开始递增。
- 已创建过的编号不得复用，即使任务被拒绝、取消或阻塞。
- 任务名使用简短、稳定、可搜索的中文名称，不把模型名写进文件名。
- 新任务必须复制 `TEMPLATE.md`，不能从旧任务复制审批记录和执行记录。

## 状态机

```text
草拟 → 等待批准 → 已批准 → 执行中 → 待验收 → 已完成
                     ↓          ↓          ↓
                  已拒绝      已阻塞      返工中
                                   ↓          ↓
                                重新规划 → 等待批准
```

### 状态定义

- `草拟`：正在收集事实和形成任务单，尚未完成审批准备。
- `等待批准`：任务链、风险、验收和确认节点已写全，等待人工决定。
- `已批准`：人工明确批准具体任务编号和范围，可以进入执行。
- `执行中`：至少一个任务阶段正在执行。
- `待验收`：阶段产出已完成，等待测试或审查结论。
- `返工中`：验收失败后按原范围修正。
- `重新规划`：范围、风险、事实或方案发生变化，需要新计划和新批准。
- `已阻塞`：缺少必要信息、权限或外部条件，记录解除条件。
- `已拒绝`：人工明确不同意执行，保留拒绝原因。
- `已取消`：需求失效或被其他任务替代，保留取消原因。
- `已完成`：所有验收、测试和独立审查证据齐全。

## 审批规则

- 只有针对具体任务编号和范围的明确批准，才能从“等待批准”进入“已批准”。
- “继续分析”“可以讨论”“先看看”不构成实施批准。
- 只批准产品设计阶段时，不得推断架构、编码、迁移或发布已经获批。
- 任务范围扩大、风险提高、数据库或权限方案实质变化、验收标准失效时，旧批准失效，必须进入“重新规划”。
- R3 任务在关键执行动作前需要第二次人工确认。

## 历史保留规则

- 不删除已产生审批、执行、测试或审查记录的任务。
- 不覆盖旧计划来隐藏范围变化；在任务内追加重新规划原因、新计划和新审批记录。
- 被取消或拒绝的任务继续保留，以便复盘路由和判断质量。
- 如果任务被拆成多个新任务，旧任务写明拆分关系，新任务分别编号。

## 执行证据规则

每个执行阶段必须记录：

- 实际执行人和能力角色。
- 实际使用模型；如替代，记录原模型、替代模型和原因。
- 实际改动或产出文件。
- 测试命令、输出结果和已知限制。
- 审查人、审查范围、审查结论和遗留风险。
- 下一状态及状态变化原因。

“已完成”只能在测试和审查证据已回写后使用。
