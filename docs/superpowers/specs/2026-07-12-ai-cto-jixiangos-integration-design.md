# 极享OS AI CTO 接入设计

> 日期：2026-07-12  
> 状态：已确认设计，待实施计划  
> 上游规范：AI项目增长仓库中的《AI CTO 通用治理设计》

## 1. 目标

把通用 AI CTO 治理规范接入极享OS研发流程。第一版通过项目内 Markdown 配置和任务文件运行，不修改应用代码，不新增数据库表，不开发管理页面。

AI CTO 收到极享OS需求后，必须先读取项目上下文和路由规则，生成待审批任务单；只有在人工批准后，才允许进入实施。

## 2. 项目文件架构

```text
docs/ai-cto/
├── AI-CTO.md
├── project-context.md
├── routing-rules.md
└── tasks/
    └── TASK-YYYYMMDD-序号-任务名.md
```

各文件职责：

- `AI-CTO.md`：AI CTO 的极享OS启动入口，规定读取顺序、决策步骤和输出位置。
- `project-context.md`：极享OS当前阶段、技术栈、核心业务链、质量门禁和高风险领域。
- `routing-rules.md`：任务类型、风险升级、能力角色和当前推荐模型映射。
- `tasks/`：所有需求的审批、执行、测试、审查和结论记录。

## 3. 启动读取顺序

AI CTO 处理需求前按以下顺序读取：

1. 通用 AI CTO 治理规范。
2. `docs/ai-cto/AI-CTO.md`。
3. `docs/ai-cto/project-context.md`。
4. `docs/ai-cto/routing-rules.md`。
5. 与需求直接相关的极享OS业务文档、代码和现有任务。

如果通用规范在另一个本地仓库不可访问，项目内启动入口应包含足以安全运行的核心约束，并明确提示当前缺少上游规范，而不是忽略治理规则。

## 4. 极享OS项目上下文

项目配置至少记录以下事实：

- 技术栈：React、Express、Prisma、MySQL。
- 当前阶段：员工稳定版验收，尚未达到正式发布标准。
- 核心业务链：线索、客户、订单审核、订单分账、售后挽回、提成核算、财务发放。
- 高风险领域：权限、数据范围、订单、财务、提成、数据库迁移、生产部署。
- 第一版业务边界以 `docs/jixiang-os-project-knowledge-base.md` 为主要依据。
- 发布结论以 `RELEASE_CHECKLIST.md` 为门禁依据。
- 质量检查以仓库当前脚本和 README 为准，不得把不存在的 lint 命令记为通过。

项目上下文属于可更新配置。项目阶段、质量命令或发布门禁发生变化时，应修改上下文，而不是修改通用治理原则。

## 5. 极享OS风险升级

以下任务最低为 R2：

- 角色权限、菜单权限、按钮权限和数据范围。
- 客户归属、线索分配和公共池领取规则。
- 正式订单、审核状态和订单入库。
- 订单分账、售后挽回分账、提成和财务状态。
- Prisma schema、迁移和跨模块状态流转。

以下任务为 R3：

- 生产部署、生产数据库或生产环境变量变更。
- 数据删除、破坏性迁移、批量修复生产数据。
- 已发放资金记录、真实资金动作或不可逆财务操作。
- 对外正式发布和影响全员使用的大版本切换。

R2 任务需要人工批准和独立审查。R3 任务除任务批准外，在关键执行动作前必须再次确认。

## 6. 初始能力角色与模型映射

| 能力角色 | 当前推荐模型 | 极享OS中的用途 |
| --- | --- | --- |
| 快速执行者 | GPT-5.5 | 低风险 UI、原型、局部代码调整 |
| 产品设计者 | GPT-5.6 Luna | 业务流程、需求澄清、验收标准 |
| 架构负责人 | GPT-5.6 Sol | 权限、订单、财务、数据库和跨模块设计 |
| 独立审查者 | GPT-5.6 Terra | 代码、安全、回归和发布审查 |

模型名称是当前建议，不是永久绑定。模型不可用时按能力角色寻找替代，并在任务单中记录替代模型和原因。

## 7. 任务文件规则

文件名采用：

```text
TASK-YYYYMMDD-序号-简短任务名.md
```

同一天序号从 `001` 开始递增。任务文件创建后不得通过覆盖或删除抹去历史；状态变化和审批记录以追加或明确修改字段的方式保留。

每份任务必须记录：

- 原始需求、战略目标和成功标准。
- 任务类型、风险等级、影响模块和明确不做。
- 分阶段任务链、能力角色、推荐模型、输入输出和依赖。
- 测试、独立审查、回滚和人工确认节点。
- 审批、执行、返工、阻塞和最终结论。

## 8. 审批与执行

初始状态为“草拟”，完成分析后进入“等待批准”。用户明确批准任务单后，记录批准时间和批准范围，状态进入“已批准”。

批准只覆盖任务单明确列出的范围。发生以下任一情况时，必须重新规划并重新审批：

- 新增业务模块或关键依赖。
- 风险等级提高。
- 数据库、权限或状态流转方案发生实质变化。
- 原验收标准无法继续适用。

执行角色完成阶段任务后，必须回写实际改动、验证命令、结果和已知限制。独立审查者不能仅复述执行者结论。

## 9. 验证门禁

AI CTO 根据任务影响范围选择仓库已有检查，完整质量基线包括：

```bash
pnpm exec tsc -b --pretty false
pnpm test
pnpm run build
pnpm exec prisma validate
pnpm exec prisma migrate status
NODE_ENV=production pnpm run prod:check
```

并非每个低风险任务都必须运行全部命令，但任务单必须事先说明选择了哪些检查及原因。涉及数据库或发布的任务不得省略相应 Prisma 或生产检查。

当前仓库没有 lint 命令，因此任何任务都不得声称 lint 已通过。若未来补充 lint，应同步更新项目上下文和验证门禁。

## 10. 错误与阻塞处理

- 缺少需求事实、权限或外部环境时，状态改为“已阻塞”并记录解除条件。
- 测试失败时保留命令和错误摘要；同一方案连续失败两次后退回 AI CTO 重新规划。
- 发现已有用户改动或与任务无关的未跟踪文件时，必须保留并绕开，不能擅自删除或覆盖。
- 生产条件、迁移基线或备份证据不足时，不得继续部署。
- 审查发现范围扩大或风险升级时，状态改为“重新规划”，旧批准失效。

## 11. 第一版交付物

后续实施阶段创建：

1. `docs/ai-cto/AI-CTO.md`。
2. `docs/ai-cto/project-context.md`。
3. `docs/ai-cto/routing-rules.md`。
4. `docs/ai-cto/tasks/README.md` 或任务模板。
5. 一个真实极享OS需求的示例待审批任务单，用于验证闭环。

示例任务应选取真实但不立即实施的需求。它只验证分析、路由、任务生成和审批边界，不借示例任务绕过后续实施审批。

## 12. 第一版不包含

- 极享OS应用中的 AI 研发指挥中心页面。
- AI CTO 数据库表、接口和状态管理代码。
- 通过 API 自动调用或切换多个模型。
- 自动发布、自动迁移或自动生产部署。
- 研发以外的 AI CEO、COO、CMO、CFO 角色。

## 13. 验收标准

1. 输入真实极享OS需求后，可生成完整的待审批任务单。
2. 权限、订单、财务、数据库和发布任务能按规则升级风险。
3. 任务未经批准不进入实施。
4. 批准范围发生变化时会重新规划和重新审批。
5. 开发、测试和独立审查均能在任务文件中留下证据。
6. 当前推荐模型不可用时，能按能力角色替代并说明原因。
7. 连续验证 5 至 10 个任务后，可汇总路由准确性、返工和阻塞原因，作为产品化依据。

## 14. 后续产品化条件

只有当 Markdown 运行中出现稳定的字段、状态、审批动作和统计需求时，才设计极享OS内的 AI 研发指挥中心。产品化前应优先验证：

- 任务分类是否稳定。
- 风险升级是否准确。
- 能力角色与模型映射是否降低返工。
- 哪些人工确认节点确实必要。
- 哪些任务状态需要数据库查询和可视化。
