# AI CTO任务模板

复制本文件创建新任务后，必须将所有示例值替换为本任务的确定内容；实际任务不得保留枚举竖线、示例名称或未解释空字段。

```yaml
任务编号: TASK-YYYYMMDD-001
任务名称: 示例任务名称
创建时间: YYYY-MM-DD HH:mm Asia/Shanghai
当前状态: 草拟

原始需求: "用户原话"
战略目标: "该任务服务的业务结果"
成功标准:
  - "可观察、可验收的结果"

任务类型: 探索验证 | 产品设计 | 常规开发 | 核心业务 | 缺陷修复 | 发布审计
风险等级: R0 | R1 | R2 | R3
风险依据:
  - "触发的具体规则"
影响模块:
  - "模块名"
明确不做:
  - "本次排除范围"
前置依赖:
  - "依赖或无"

任务链:
  - 阶段编号: 1
    阶段名称: "阶段名称"
    能力角色: "能力角色"
    推荐模型: "当前推荐模型"
    替代模型记录: "未替代，或原模型、替代模型及原因"
    输入:
      - "输入"
    预期输出:
      - "输出"
    验收标准:
      - "验收标准"
    依赖阶段: "无或阶段编号"

测试要求:
  - "命令、检查项及选择原因"
审查要求:
  - "审查角色和范围"
回滚方案:
  - "如何恢复；纯文档任务写明 Git revert"
人工确认节点:
  - "批准实施或 R3 关键动作"

审批记录:
  - 时间: "未审批"
    审批人: "未审批"
    决定: "等待批准"
    批准范围: "无"

执行记录: []
最终结论: "未执行"
```

## 字段解释

- `任务类型` 必须选择一个确定值，不得保留枚举选项。
- `风险等级` 必须引用 `routing-rules.md` 的具体触发规则。
- `任务链` 按依赖顺序列出阶段；后续阶段不能使用未获批的前置假设。
- `测试要求` 写明实际要运行的命令或检查；不适用时说明原因。
- `审查要求` 写明独立审查角色和范围；R2、R3 不能写“无需审查”。
- `人工确认节点` 区分任务批准、范围变化批准和 R3 关键动作二次确认。
- “等待批准”的任务不得填写执行结果，`执行记录` 保持空数组，`最终结论` 为“未执行”。
- 任务批准后，追加审批记录，不删除“未审批”历史。
