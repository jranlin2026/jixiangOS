# 极享OS 项目知识库

> 更新时间：2026-06-30  
> 适用范围：极享OS CRM/企业运营系统第一版产品、业务、权限、财务、售后、交付与部署知识沉淀。

## 1. 项目定位

极享OS 是面向公司内部使用的 AI 企业运营系统。第一版目标不是做一个大而全的 ERP，而是把销售型公司的关键业务链路跑通：

```text
线索进入 -> 客户沉淀 -> 订单审核 -> 订单分账 -> 售后挽回 -> 提成核算 -> 员工查看 -> 财务发放
```

系统的核心价值是：

- 让业务数据统一沉淀，不再散落在微信、表格和个人手里。
- 让订单、提成、售后挽回这些高争议流程有依据、有记录、可追溯。
- 让员工能看清楚自己该拿多少钱，财务能看清楚该发多少钱。
- 让管理层能看到业务流转、订单阻塞、提成风险和交付状态。

## 2. 第一性原理

做这个系统的第一性原理是：公司业务最终要回答“钱从哪里来、归谁负责、应该分给谁、现在卡在哪里”。

因此系统所有模块都围绕四类事实设计：

| 事实 | 系统承载 | 关键问题 |
| --- | --- | --- |
| 客户事实 | 线索、客户、跟进、动态 | 谁带来的客户？谁在跟？是否进入成交？ |
| 收入事实 | 订单、订单审核台、收支流水 | 这笔钱是否真实到账？是否可确认为收入？ |
| 责任事实 | 负责人、录入人、销售、售后、交付人员 | 谁贡献了这笔业务？谁该处理下一步？ |
| 分配事实 | 订单分账、售后挽回分账、员工提成 | 按什么规则算？谁拿多少？是否已确认和发放？ |

第一版的取舍原则：

- 能简单人工处理的复杂异常，先不放进系统。
- 钱和权限相关的动作必须有状态、有记录、有审核边界。
- 售后、财务、销售的入口要分清，避免一个模块承载所有事情。
- 员工端只看与自己相关的信息，管理/财务端按权限和数据范围查看。

## 4. 线索模块

线索模块用于承接市场、官网、表格导入、人工新增等渠道来的潜在客户。

关键规则：

- 线索不等于客户，线索只有进入正式客户后才沉淀为客户资产。
- 线索列表受数据权限控制，普通员工默认只能看本人范围。
- 超级管理员或拥有全部数据范围的角色可以看全部线索。
- 批量导入 Excel 时，需要用系统模板字段，重复手机号会被拦截并展示失败原因。

线索常见动作：

- 新建线索
- 批量入库
- 下载 Excel 模板
- 分配销售
- 跟进并加入客户
- 查看入库情况

## 5. 客户模块

客户模块是系统的业务资产库。客户可以由线索转入，也可以由有权限的人员直接创建。

核心信息：

- 客户基本资料：姓名、公司、手机、微信、来源、行业、城市等。
- 客户归属：公司资源、个人资源等。
- 客户角色关系：销售负责人、线索录入人、资源录入人等。
- 客户动态：跟进记录、订单记录、流转记录、附件记录。

当前设计原则：

- 售后人员不因为要提交售后挽回订单而获得客户列表权限。
- 售后挽回订单中的客户姓名、手机号、微信只是普通业务字段，不自动创建客户档案。
- 客户资料详情里不再保留没有真实业务意义的字段。

关于“客户画像”和“AI名片”：

- 客户画像：偏内部经营分析，用于总结客户来源、需求、成交潜力、历史订单、跟进状态。
- AI名片：偏对外或销售辅助表达，用 AI 把客户/产品/服务信息整理成可展示内容。
- 第一版如果使用频率不高，可以先弱化为客户详情里的辅助能力，不作为核心权限入口。

## 6. 订单模块

订单模块只处理正式销售订单，也就是客户购买产品后形成的真实收入单据。

流程：

```text
提交订单申请 -> 订单审核台 -> 财务/管理员审核 -> 订单列表入库 -> 订单分账
```

订单列表和订单审核台的边界：

- 订单审核台：处理待审核订单申请，支持通过、退回修改、驳回。
- 订单列表：只展示已审核入库的正式订单。
- 新提交的订单不应该直接出现在订单列表。

第一版取消系统内退款功能：

- 订单不能发起退款。
- 不显示退款状态、退款金额、退款冲销等入口。
- 真实发生退款时，财务线下处理，系统第一版只在订单分账中通过撤回或手动调整处理未发放前错误。

## 7. 交付模块

交付模块只处理“确实需要交付”的订单。

当前规则：

- 交付阶段由系统设置 -> 产品配置中配置。
- 新建产品默认交付阶段为空。
- 如果产品的交付阶段为空，该产品对应订单不进入交付工单台。
- 系统不再提供默认兜底交付模板。
- 已有预设产品的默认交付阶段已清空，是否需要交付由产品配置决定。

交付工单状态：

- 待开始
- 交付中
- 超期
- 阻塞
- 待验收
- 已完成

交付步骤原则：

- 每个产品可以配置自己的步骤。
- 不支持“跳过步骤”作为常规能力。
- 支持完成当前步骤。
- 需要返回上一步时，应作为交付纠错能力，而不是随意跳转流程。

## 8. 售后服务模块

售后服务模块第一版只做售后挽回，不做退款。

模块包含两个 Tab：

```text
售后服务
├─ 售后挽回订单列表
└─ 售后挽回订单审核台
```

### 8.1 售后挽回订单是什么

售后挽回订单不是正式订单，不进入订单中心，也不计算正式销售业绩。

它的作用是记录第三方平台订单或外部订单发生售后挽回后，作为提成核算依据。

典型场景：

- 客户在第三方平台购买。
- 售后人员做了挽回或二次成交。
- 这笔业务不在 CRM 正式订单库里。
- 但员工需要基于挽回事实拿提成。

### 8.2 售后挽回订单流程

```text
新建售后挽回订单
      ↓
售后挽回订单审核台
      ↓ 财务/管理员审核
审核通过
      ↓
售后挽回订单列表
      ↓
财务中心 -> 售后挽回分账
      ↓
员工我的提成
```

重要边界：

- 新建后先进入审核台。
- 审核通过后才进入售后挽回订单列表。
- 审核通过后才进入财务中心的售后挽回分账待处理范围。
- 退回修改的单据仍属于审核台处理范围，不放到正式列表里当已入库数据。

### 8.3 新建售后挽回订单字段

字段包括：

- 客户姓名
- 客户手机号
- 客户微信
- 第三方平台订单号
- 来源平台
- 原购买产品
- 原付款金额
- 挽回成交金额
- 挽回人员
- 收款凭证附件
- 聊天记录截图附件
- 备注

已取消的设计：

- 不匹配客户。
- 不自动创建售后临时客户档案。
- 不需要协同人员字段。
- 不做退款状态。

## 9. 财务中心

财务中心第一版聚焦提成和分账，不做复杂退款冲销。

当前 Tab：

```text
财务中心
├─ 我的提成
├─ 订单分账
├─ 售后挽回分账
├─ 员工提成月报
├─ 收支流水
└─ 提成规则
```

已移除：

- 财务总览
- 退款冲销
- 退款审核
- 已发放后系统内冲销

### 9.1 订单分账

订单审核通过后进入订单分账。

订单分账状态：

- 待处理
- 待确认
- 待发放
- 已发放
- 已撤回

状态判断原则：

- 没有明确提成规则、没有提成方案、没有自定义金额、人员未分配、金额为 0 且原因是规则未命中时，显示为待处理。
- 分账角色、人员、方案或自定义金额、计算说明完整后，才进入待确认。
- 待处理不能直接确认，需要财务先处理分账。
- 待确认可以确认分账，确认后进入待发放。
- 已发放不允许撤回，第一版不做系统内冲销。

### 9.2 售后挽回分账

售后挽回分账和订单分账的操作逻辑保持一致，只是业务来源字段不同。

来源：

- 订单分账来源于正式订单。
- 售后挽回分账来源于售后挽回订单。

售后挽回分账不应该混入订单分账列表，两者分开处理；员工端“我的提成”可以合并展示，但要区分来源。

### 9.3 撤回规则

第一版保留“未发放前撤回”：

- 待确认可以撤回。
- 待发放可以撤回。
- 撤回必须填写原因。
- 已撤回不进入发放、不计入待发放金额。
- 已发放不支持撤回，提示财务线下处理。

### 9.4 提成方案与提成规则

系统把“提成计算方式”抽象为提成方案，再由规则选择方案。

提成方案支持：

- 固定金额
- 固定比例
- 销售月累计阶梯
- 自定义金额

设计原则：

- 不在每条规则里重复配置计算方式和数值。
- 提成规则只选择某个提成方案。
- 分账时可以选择方案，也可以用自定义金额。
- 阶梯提成只统计阶梯提成订单的 GMV，不混入普通提成订单。

### 9.5 我的提成

我的提成面向员工本人，不是财务审核台。

设计原则：

- 顶部只展示本人提成汇总。
- 订单明细要适合订单量增长，不能无限堆卡片。
- 同一提成角色下，普通提成订单和阶梯提成订单要分开。
- 销售阶梯视图只展示阶梯订单的 GMV、当前档位和预计提成。
- 普通提成视图只展示订单、实付金额、计算说明、提成金额和状态。
- 员工要能看清“这笔提成来自正式订单分账，还是售后挽回分账”。

## 10. 权限模型

权限分为两层：

```text
菜单/功能权限：能不能看到入口、能不能操作
数据范围权限：能看到谁的数据
```

数据范围：

- 本人
- 本部门
- 全部

当前数据范围域：

- 线索数据
- 客户数据
- 订单数据
- 订单审核台数据
- 售后挽回订单数据
- 售后挽回订单审核台数据

### 10.1 售后服务权限

售后服务权限拆成一级列表：

- 售后挽回订单列表
- 售后挽回订单审核操作
- 新增售后挽回订单
- 编辑售后挽回订单
- 删除售后挽回订单
- 售后挽回订单修改记录

权限边界：

- 售后员工可以新增售后挽回订单，但不一定能审核。
- 是否能进审核台由“售后挽回订单审核操作”控制。
- 审核台能看哪些单据，由“售后挽回订单审核台数据”控制。
- 超级管理员默认全部权限和全部数据范围。

### 10.2 财务角色

财务角色通常需要：

- 财务中心/订单分账
- 财务中心/售后挽回分账
- 财务中心/员工提成月报
- 财务中心/收支流水
- 财务中心/提成规则
- 售后服务/售后挽回订单审核操作
- 订单审核台数据：全部
- 售后挽回订单审核台数据：全部

### 10.3 售后角色

售后角色通常需要：

- 售后挽回订单列表
- 新增售后挽回订单
- 编辑本人退回修改的售后挽回订单
- 售后挽回订单修改记录
- 售后挽回订单数据：本人或本部门

一般不需要：

- 客户列表
- 订单审核台
- 售后挽回审核操作
- 财务分账操作

## 11. 数据与持久化

当前项目已接入：

- React + Vite 前端
- Express 后端
- Prisma
- MySQL
- 业务数据共享存储

本地开发默认：

```text
前端：http://127.0.0.1:3000 或 3002
后端：http://127.0.0.1:3001
数据库：MySQL
```

关键命令：

```powershell
npm.cmd run dev
npm.cmd run build
npm.cmd run db:generate
npm.cmd run db:push
npm.cmd run db:seed
```

注意：

- 第一阶段不是所有业务都已经拆成结构化 MySQL 表。
- 目前大量业务数据通过后端共享存储同步到 MySQL，避免只存在单个浏览器 localStorage。
- 售后挽回订单、角色、权限、订单、交付、分账等都需要跨账号可见，因此必须保证写入后端。
- 切换账号前需要确保业务写入完成，系统已加入写入等待机制。

## 12. 部署知识

第一版推荐部署方式：

```text
阿里云服务器 + 域名 + HTTPS + 登录账号 + MySQL + 每日备份
```

推荐配置：

- Ubuntu 22.04/24.04
- 2 核 4G 起步
- 80G 磁盘起步
- 安全组只开放 22、80、443
- MySQL 3306 不对公网开放

部署要点：

- 使用 GitHub 仓库 `jranlin2026/jixiangOS`
- 分支：`codex/core-crm-polish`
- 使用 Nginx 代理前端和后端 API
- 使用 Certbot 配置 HTTPS
- 使用 PM2 守护后端进程
- 使用 cron 每日备份 MySQL

验收标准：

- 域名 HTTPS 可访问
- 管理员可登录
- 首页、线索、客户、订单、交付、售后、财务、设置可打开
- 后端健康检查正常
- MySQL 数据可持久化
- 每日备份目录生成 `.sql.gz`

## 14. 当前版本的核心决策记录

### 14.1 财务中心简化

保留：

- 我的提成
- 订单分账
- 售后挽回分账
- 员工提成月报
- 收支流水
- 提成规则

移除：

- 财务总览
- 退款冲销
- 已发放后系统内冲销

原因：第一版先跑通正常收款和提成链路，复杂退款追回后续单独设计。

### 14.2 售后服务独立

售后挽回不是正式订单，所以不放订单中心。

售后服务只承载售后挽回事实，财务审核后进入售后挽回分账。

### 14.3 交付阶段由产品决定

没有交付阶段配置的产品，就不需要交付工单。

系统不再给产品默认套交付流程，避免“不该交付的订单也出现在交付台”。

### 14.4 提成规则改为方案化

提成计算方式不再每次在规则里重复填，而是先配置提成方案，再由规则和分账选择方案。

这样财务能理解：

```text
谁拿钱 -> 用哪个方案 -> 算出多少钱 -> 是否确认 -> 是否发放
```

## 15. 常见问题

### Q1：为什么售后人员不能直接看客户列表？

因为售后挽回只需要提交业务事实，不需要开放客户资产池。客户列表权限过大，会让售后看到与自己无关的客户数据。

### Q2：为什么售后挽回订单不是订单？

因为它不是 CRM 内正式成交订单，不进入正式销售业绩；它只是第三方平台或外部订单挽回后的提成核算依据。

### Q3：为什么退款功能取消？

第一版退款会牵涉冲销、追回、已发放扣回、订单收入修正等复杂链路。当前先用财务线下处理真实退款，系统只处理未发放前的撤回。

### Q4：为什么订单审核通过后有时是待处理，不是待确认？

如果没有命中明确提成规则、没有方案、没有人员、金额为 0 且原因是规则未命中，就不能直接确认，必须先由财务处理分账。

### Q5：为什么新建产品默认没有交付阶段？

因为不是所有产品都需要交付。只有在产品配置里明确设置了交付阶段，该产品订单才进入交付工单台。

### Q6：为什么换账号后数据要还能看到？

因为系统要用于公司内部协作，不能只存在单个浏览器状态。售后提交、财务审核、管理员查看都必须基于共享持久化数据。
