# 客户标签安全删除设计

## 目标

让管理员能清理从未被业务数据引用的标签和空分组，同时保留所有已使用标签的历史可追溯性。

## 范围

- 标签：保留编辑、排序、合并、停用；为零引用标签提供永久删除。
- 分组：保留编辑、合并、停用；为没有标签的分组提供永久删除。
- 已停用标签和分组默认不显示，管理员可主动查看并恢复。
- 修复现有标签分组接口把作用域静默写死为“客户”的问题。

不改动客户、线索或历史标签迁移的数据模型；不为已使用的标签或非空分组提供删除入口。

## 管理规则

| 对象 | 条件 | 可执行操作 |
| --- | --- | --- |
| 标签 | 使用次数大于 0 | 编辑、合并、停用或启用；禁止删除 |
| 标签 | 使用次数为 0 | 编辑、停用或启用、永久删除 |
| 分组 | 含有标签 | 编辑、合并、停用或启用；禁止删除 |
| 分组 | 没有标签 | 编辑、停用或启用、永久删除 |

删除前均需显示不可恢复的二次确认。服务端再次验证条件，不能只依赖页面按钮是否可见。删除成功后刷新目录与客户/线索标签缓存。

## 页面交互

标签行和分组行采用“更多”菜单收纳低频操作。主列表继续突出编辑、合并和停用；菜单内仅在满足删除条件时展示红色“删除”。

页面默认请求并渲染启用目录。顶部提供“显示已停用”开关；打开后，停用项以弱化样式出现并可启用、编辑或治理。未显示停用项时，停用项目不占用日常配置空间。

删除确认弹窗分别说明：

- 标签：仅能删除“使用 0 次”的标签；确认后不可恢复。
- 分组：仅能删除“没有标签”的分组；确认后不可恢复。

若并发操作使对象已被引用或分组已包含标签，接口返回冲突信息，页面刷新目录并提示管理员使用合并或停用。

## 服务与数据流

客户标签服务新增 `deleteTag` 与 `deleteGroup`：二者仅限超级管理员，在事务中重新读取目录、验证对象存在和删除前提，再删除相应配置记录并写入目录审计记录。

服务端继续支持 `customer`、`lead` 与 `both` 三种分组作用域。创建或更新分组时保留调用方提交的合法作用域；若作用域或单选/多选模式变化，先用真实客户和线索引用进行模拟校验，存在冲突时原子拒绝更新。该校验恢复当前基线测试所要求的失败语义。

前端 API 层新增对应的 `DELETE` 调用；设置页按服务端目录真实状态决定菜单可见性。页面不在客户端自行推断最终安全性。

## 验收标准

1. 使用次数为 0 的标签可经二次确认删除；有引用的标签没有删除入口，服务端也返回冲突。
2. 空分组可经二次确认删除；含标签的分组没有删除入口，服务端也返回冲突。
3. 合并仍会迁移引用并停用源标签或源分组。
4. 默认状态不显示停用项；开启后可看到并重新启用它们。
5. 将已有客户引用的分组从 `both` 改为 `lead` 返回冲突且不写入；现有客户标签服务测试恢复通过。
6. 相关 API、服务、页面测试及完整测试套件通过。
