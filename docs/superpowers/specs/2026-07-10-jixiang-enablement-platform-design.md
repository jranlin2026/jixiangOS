# 极享公司赋能中台设计方案

> 设计日期：2026-07-10
> 适用项目：极享OS
> 设计状态：已完成业务确认，等待书面方案复核
> 第一阶段范围：极享内部员工的通用新人7天上岗闭环

## 1. 结论

极享公司赋能中台不单独创建第二个产品、代码仓库或数据库。它作为极享OS内部一个边界清晰、未来可拆分的业务域建设。

```text
极享OS
├── 经营管理系统
├── 公司赋能中台
└── 共享AI能力底座
```

三者的职责分别是：

- 经营管理系统记录线索、客户、订单、交付、售后、财务和经营结果。
- 公司赋能中台负责企业知识发布、员工学习、考试、实战任务、主管确认和培训运营。
- 共享AI能力底座同时服务经营分析和知识学习，但两类场景使用不同的知识范围、提示词与权限。

第一阶段只服务极享内部员工，以“全体新人通用入职”为第一个端到端闭环。后续再扩展到岗位培训和在岗工作提效。

## 2. 为什么不单独开发项目

极享OS已经具备员工账号、部门、岗位、主管、角色权限、MySQL、文件服务、DeepSeek配置、经营数据和业务待办。新人学习与这些能力天然相关：

- 新员工创建后需要自动分配学习路径。
- 部门负责人需要审核知识并查看直属员工进度。
- 主管需要完成实战任务评审。
- 知识权限依赖部门、岗位和角色。
- 培训结果未来需要与销售、交付和经营数据关联。

如果现在单独开发中台，将重复建设登录、组织、权限、员工数据和系统设置，还要长期处理两个系统之间的身份与数据同步。

因此采用模块化单体：产品和部署保持一体，中台拥有独立前端路由、后端接口、服务层和数据表。只有出现以下条件时才评估拆分：

1. 中台需要面向外部企业提供多租户服务。
2. 中台由独立团队按不同节奏发布和运维。
3. 除极享OS外，有两个以上系统需要独立调用中台。
4. 知识索引或文件处理需要独立扩缩容。

## 3. 已确认的产品边界

### 3.1 第一阶段用户

- 仅极享内部员工。
- 不包含代理商、服务商和外部客户。
- 第一条学习路径面向全体新人，不区分销售、流量、交付等岗位。

### 3.2 第一阶段完成标准

新人只有同时满足以下条件，才能被标记为“已通过上岗”：

- 必修学习完成率达到100%。
- 正式考试达到配置的合格分，默认80分。
- 所有必修实战任务已提交。
- 主管完成最终确认并选择通过。

AI不拥有最终上岗决定权。

### 3.3 知识源与运行平台

- WPS/Obsidian是知识编写与沉淀来源。
- 极享OS是正式发布、权限控制、员工学习、考试、任务、AI调用和数据追踪平台。
- 第一阶段使用管理员人工导入，不做WPS双向同步。
- 部门负责人负责内容正确性审核，知识库管理员负责版本治理与正式发布。

## 4. 总体产品架构

```text
极享OS
├── 经营管理系统
│   ├── 驾驶舱
│   ├── 线索与客户
│   ├── 订单
│   ├── 交付与售后
│   └── 财务与提成
│
├── 公司赋能中台
│   ├── 中台首页
│   ├── 企业知识中心
│   ├── 学习培训中心
│   ├── 考试与实战
│   ├── AI导师
│   ├── 内容治理后台
│   └── 培训运营看板
│
└── 共享AI能力底座
    ├── AI模型网关
    ├── 权限过滤
    ├── 企业知识检索
    ├── 提示词与场景编排
    └── 来源引用与审计
```

### 4.1 AI助手与AI导师的边界

- 现有AI助手继续分析极享OS经营数据、风险、任务和经营问题。
- AI导师只基于已正式发布且员工有权限查看的企业知识，提供讲解、问答、练习和薄弱项建议。
- 两者共用模型配置与调用服务，不共用未经筛选的上下文。
- AI导师的对话和引用记录属于赋能中台数据，不混入经营AI会话。

## 5. 新人7天上岗流程

### 5.1 主流程

```text
创建员工账号
  -> 自动分配通用入职路径
  -> 按日完成学习和AI辅导
  -> 完成阶段练习
  -> 参加正式考试
  -> 提交实战任务
  -> 主管评审
  -> 通过上岗或退回补学
```

### 5.2 状态机

```text
待开始
  -> 学习中
  -> 待考试
  -> 待实战
  -> 待主管确认
  -> 已通过

考试不合格或主管退回
  -> 补学中
  -> 重新考试或重新提交
```

状态转换由明确的完成条件驱动，不允许前端直接把新人改成“已通过”。

### 5.3 默认7天内容

| 天数 | 主题 | 学习成果 |
|---|---|---|
| 第1天 | 公司认知与战略方向 | 能准确介绍极享、核心业务和发展方向 |
| 第2天 | 产品矩阵与客户价值 | 知道公司卖什么、为谁解决什么问题 |
| 第3天 | 部门、岗位与协作关系 | 知道业务角色、责任边界和交接对象 |
| 第4天 | 公司前端到后端业务闭环 | 理解流量、销售、订单、交付、售后和财务 |
| 第5天 | 制度、合规与信息安全 | 知道公司规则、宣传红线和敏感信息边界 |
| 第6天 | 极享OS与AI协作 | 能查知识、处理任务、使用AI导师和基础系统功能 |
| 第7天 | 综合考试与实战 | 验证理解、应用能力和上岗准备度 |

“7天”是默认学习路径配置，不写死在业务代码中。管理员可以调整天数、顺序、必修项和合格分。

默认日程以员工学习路径的开始日期为第1天，按自然日计算。培训管理员可以暂停、恢复或调整个人截止日期；所有调整保留操作记录。

## 6. 第一阶段功能模块

### 6.1 中台首页

按当前用户角色展示：

- 新人：今天学什么、进度、待考试、待提交、截止时间和AI导师入口。
- 主管：待评审任务、逾期新人和直属员工进度。
- 部门负责人：待审核知识和部门薄弱项。
- 知识库管理员：待发布、解析失败、版本冲突和知识缺口。
- 培训管理员：全公司新人完成情况、考试和逾期情况。

### 6.2 企业知识中心

首批知识分类：

- 公司认知
- 项目与产品
- 部门与岗位
- 销售与流量
- 交付与售后
- 制度与SOP
- AI能力与使用规范

员工可以按权限浏览、搜索和查看当前有效版本。历史版本不进入普通员工默认搜索。

### 6.3 学习培训中心

- 课程管理
- 学习路径配置
- 路径项目排序与按日安排
- 必修、选修标记
- 员工学习记录
- 重新学习任务

### 6.4 考试与实战

- 正式题库由负责人审核后使用。
- AI生成题只用于练习，不自动进入正式题库。
- 第一阶段正式题型支持单选、多选和判断题；需要主管判断的开放题放入实战任务。
- 考试支持合格分、次数和重考规则配置。
- 实战任务支持文字、附件和系统操作证明。
- 主管可以通过、退回并填写评语。

### 6.5 AI导师

- 基于当前课程或员工问题进行带来源问答。
- 用更容易理解的语言重新讲解知识。
- 生成日常练习题和追问。
- 根据练习与正式考试结果提示薄弱知识点。
- 给出复习顺序和相关课程入口。
- 知识不足时生成知识缺口，不编造公司规则。

### 6.6 内容治理后台

- 导入WPS/Obsidian资料。
- 部门审核。
- 管理员发布、下线和版本回滚入口。
- 查看文件哈希、解析状态、有效期和权限。
- 处理知识冲突与知识缺口。

### 6.7 培训运营看板

- 分配人数与开始人数
- 完成人数与完成率
- 平均完成时长
- 考试通过率与重考情况
- 实战任务通过和退回情况
- 逾期人员
- 高频薄弱知识点
- AI无法回答的高频问题

系统提供指标和筛选能力；具体经营目标值由培训管理员按阶段配置，不硬编码在产品中。

## 7. 知识发布与版本治理

### 7.1 发布流程

```text
WPS/Obsidian编写
  -> 管理员导入极享OS
  -> 系统校验元数据与文件
  -> 部门负责人审核
  -> 知识库管理员发布
  -> 生成不可变版本
  -> 文本提取与切分
  -> 建立可检索索引
  -> 加入课程或学习路径
```

服务器不直接读取管理员Mac上的WPS本地路径。第一阶段通过浏览器上传文件或资料包；源路径仅作为可选溯源信息保存。

### 7.2 内容状态

```text
草稿 -> 待审核 -> 已审核 -> 发布处理中 -> 当前有效
                      |          |
                      |          -> 发布失败
                      -> 驳回修改

当前有效 -> 已下线 -> 历史归档
```

只有“当前有效”且在生效日期范围内的版本能进入普通员工搜索和AI检索。

### 7.3 版本规则

- 每次正式发布产生新的不可变版本，不覆盖旧版。
- 文档维护一个稳定入口和一个当前版本指针。
- 学习记录保存实际学习的知识版本ID。
- 旧版本下线后立即退出AI检索，但历史学习、考试和审计记录保留。
- 价格、权益、提成、合同与制度需要生效日期。
- 发布重大更新时，管理员可以给指定员工生成重新学习任务。
- 重复导入使用SHA-256识别，管理员选择忽略或创建新版本。

## 8. 文件与内容处理

### 8.1 第一阶段支持

- Markdown：提取标题、正文、层级与元数据，直接进入知识索引。
- PDF：保存原文件并提取文本；解析成功后进入知识索引。
- PPT/PPTX：作为课程附件保存，必须补充课程说明或讲师稿后才能进入AI检索。
- 视频：作为课程附件保存，必须提供字幕或转写文本后才能进入AI检索。
- 图片与截图：必须填写说明、来源和授权状态；仅图片本身不进入公司事实索引。

### 8.2 私有文件访问

赋能中台附件不能直接放在无需鉴权的公开静态目录。下载和预览必须经过登录、知识权限和文件权限校验。

## 9. AI知识检索

### 9.1 回答流程

```text
员工提问
  -> 读取当前员工身份、部门、角色与权限
  -> 过滤可见的当前有效知识版本
  -> 检索相关知识片段
  -> 组装受控上下文
  -> 调用DeepSeek
  -> 返回答案、来源、版本、更新时间和相关入口
  -> 记录引用与知识缺口
```

权限过滤必须发生在知识内容发送给模型之前。

### 9.2 第一阶段检索方案

第一阶段知识量有限，使用MySQL保存知识片段、标题、标签和关键词，先实现可替换的 `KnowledgeSearchProvider` 接口。默认实现采用元数据过滤、中文关键词匹配和结果排序，不在第一阶段引入独立向量数据库。

未来切换到向量或混合检索时，只替换搜索实现，不改变知识版本、权限、课程和学习记录。

### 9.3 AI回答规则

- 只能引用员工有权访问的当前有效知识。
- 回答必须返回文档标题、版本、更新时间和对应片段。
- 找不到充分来源时明确说明没有正式依据。
- 多个当前版本冲突时停止给出确定结论，并生成冲突记录。
- 财务、提成、客户数据和管理层资料按角色隔离。
- AI不可创建、发布或批准公司制度。

## 10. 与现有极享OS的技术集成

### 10.1 前端边界

新增顶级路由 `/enablement`，中台页面放在独立目录中，不继续堆入 `AIAssistant`：

```text
src/pages/Enablement/
├── Home/
├── Knowledge/
├── Learning/
├── Exams/
├── Tasks/
├── Reviews/
├── Publishing/
└── Analytics/
```

侧边栏新增“赋能中台”，子入口按权限显示。

### 10.2 后端边界

新增统一命名空间 `/api/enablement`：

```text
/api/enablement/knowledge
/api/enablement/publications
/api/enablement/learning-paths
/api/enablement/enrollments
/api/enablement/exams
/api/enablement/tasks
/api/enablement/reviews
/api/enablement/ai
/api/enablement/analytics
```

当前 `server/index.ts` 内的DeepSeek调用应提取为共享AI网关服务，使现有经营AI与新AI导师复用连接配置，同时保持提示词和上下文隔离。

### 10.3 新员工自动分配

员工账号创建成功后，组织服务调用赋能中台的幂等分配服务：

```text
assignDefaultOnboarding(userId)
```

重复调用不会重复创建学习任务。默认路径按“全体新人”受众规则匹配。

第一阶段的“主管”定义为该次学习分配中指定的评审人。系统创建分配时默认使用员工所属部门的 `managerId`，并把评审人保存到 `Enrollment.reviewerUserId`；培训管理员可以重新指派。这样不依赖当前用户表中不存在的直属主管字段，也能保留当时的评审责任人。

## 11. 核心数据模型

关键学习与知识记录需要版本和审计，不使用通用 `BusinessRecord` 作为唯一存储。

### 11.1 知识治理

| 对象 | 核心职责 |
|---|---|
| KnowledgeDocument | 稳定知识入口、分类、负责人和当前版本 |
| KnowledgeVersion | 不可变发布版本、文本、哈希、生效时间和状态 |
| KnowledgeAttachment | 原文件、附件类型、私有存储位置和授权信息 |
| KnowledgeVisibility | 文档可见部门、角色、岗位和敏感级别 |
| ContentReview | 审核人、决定、意见和时间 |
| KnowledgeChunk | AI检索片段、标题、顺序、标签和版本关系 |
| KnowledgeGap | 无法回答、冲突或缺失知识的反馈记录 |

### 11.2 学习与上岗

| 对象 | 核心职责 |
|---|---|
| Course / CourseItem | 一门课程及其知识、附件和内容顺序 |
| LearningPath | 学习路径、受众、默认天数和通过规则 |
| LearningPathItem | 课程、考试、任务的顺序、日期偏移和必修属性 |
| Enrollment | 员工与学习路径的分配、状态、起止时间和指定评审人 |
| LearningProgress | 员工对具体项目和知识版本的完成记录 |
| QuestionBank / Question | 经审核的正式题库与题目 |
| Exam / ExamAttempt | 考试配置、得分、次数和结果 |
| PracticeTask / TaskSubmission | 实战任务要求与员工提交 |
| SupervisorReview | 主管通过、退回、评语和时间 |

### 11.3 AI导师与审计

| 对象 | 核心职责 |
|---|---|
| MentorConversation / MentorMessage | AI导师会话、问题、答案和用户范围 |
| AiCitation | 每次回答实际使用的知识文档、版本和片段 |
| AiFeedback | 员工对答案的有用、错误或缺失反馈 |

所有关键表保留创建人、更新时间和必要的审计字段。

## 12. 权限设计

新增权限至少包括：

- 查看企业知识
- 参加学习与考试
- 提交实战任务
- 查看直属员工学习情况
- 评审直属员工实战任务
- 审核部门知识
- 发布与下线公司知识
- 管理课程与学习路径
- 管理正式题库
- 查看部门培训数据
- 查看全公司培训数据
- 查看敏感知识

数据范围继续复用现有极享OS角色与部门规则。普通员工只能查看自己的学习记录；主管默认查看直属员工；部门负责人查看本部门；培训管理员按授权查看全公司。

## 13. 异常与降级处理

| 异常 | 系统行为 |
|---|---|
| 缺少负责人、分类、权限或审核人 | 阻止进入审核或发布 |
| 文件解析失败 | 保留原文件，标记解析失败，不进入AI索引 |
| 导入重复文件 | 显示已有版本，选择忽略或创建新版本 |
| 当前知识发生冲突 | 阻止确定性AI回答，进入冲突处理队列 |
| AI找不到充分依据 | 明确说明无正式依据并生成知识缺口 |
| AI服务不可用 | 阅读、学习、考试、任务和主管确认继续工作 |
| 主管离职或停用 | 转交部门负责人或培训管理员处理 |
| 文档下线 | 立即退出普通搜索和AI检索，历史记录保留 |
| 重大制度更新 | 可生成指定人员重新学习任务 |
| 发布索引构建失败 | 不切换当前版本，继续使用上一有效版本 |

发布流程必须具备原子性：新版本只有在原文件保存、元数据校验和知识索引全部成功后，才切换为当前有效版本。

## 14. 测试与验收

### 14.1 核心验收场景

1. 创建新人账号后自动获得通用入职路径，重复触发不会重复分配。
2. 新人只能看到自己有权限的知识和学习内容。
3. 管理员能导入WPS资料，完成审核、发布和新版本更新。
4. AI导师能基于当前有效知识回答，并展示来源、版本和更新时间。
5. 新人可以完成课程、正式考试和实战任务。
6. 未达到合格分或未完成必修任务时，不能进入主管最终通过。
7. 主管可以通过或退回并留下意见。
8. 所有门槛满足后，系统才能把新人标记为已通过上岗。
9. 培训管理员能查看完成率、逾期人员、考试和薄弱知识点。
10. 旧版本下线后不再被AI引用，但历史学习记录仍可追溯。
11. AI服务故障时，基本学习、考试、任务和评审仍可使用。
12. 普通员工无法读取财务、提成、客户敏感和管理层知识。

### 14.2 测试层级

- 单元测试：状态机、合格门槛、权限过滤、版本切换、重复分配和检索排序。
- 服务集成测试：导入到发布、员工分配到上岗、版本下线到AI检索退出。
- API权限测试：普通员工、主管、部门负责人、知识管理员和培训管理员。
- 浏览器端到端测试：新人完整通关、主管退回再提交、管理员发布新版本。
- 降级测试：AI不可用、解析失败、索引失败和主管停用。
- 安全测试：跨部门敏感知识、私有附件下载和AI上下文泄漏。

## 15. 第一阶段不做

- 外部代理商、服务商或客户访问。
- 第二套登录、组织、权限和数据库。
- 极享OS内复杂在线文档编辑器。
- WPS双向或实时自动同步。
- 完整直播授课和视频平台。
- AI自动决定员工是否上岗。
- 销售、流量、交付全部岗位课程。
- 积分商城、勋章和排行榜。
- 所有历史资料一次性导入AI。
- 独立向量数据库。

## 16. 分阶段演进

### 阶段一：全体新人通用入职

完成知识发布、学习路径、考试、实战任务、主管确认、AI导师和运营看板的完整闭环。

### 阶段二：岗位学习路径

增加销售、流量、交付、售后、财务等岗位路径，并复用第一阶段底座。

### 阶段三：在岗持续赋能

支持产品更新、制度更新、专项训练、重新学习和知识缺口闭环。

### 阶段四：AI进入岗位工作

把企业知识与经营数据组合，提供销售诊断、内容生成、交付助手、合规检查和经营分析。

### 阶段五：学习与业务结果联动

分析培训行为与跟进速度、成交、交付质量、退款和其他经营结果之间的关系，使中台从培训系统升级为持续提效系统。

## 17. 最终产品定义

极享公司赋能中台是一套建立在极享OS组织、权限和业务数据之上的内部学习与知识执行系统。它把WPS/Obsidian中的知识经过审核和版本治理，变成员工可学习、AI可调用、主管可验收、管理层可复盘的公司能力。

第一阶段的唯一业务闭环是：

```text
知识发布
  -> 新人学习
  -> AI辅导
  -> 正式考试
  -> 实战任务
  -> 主管确认
  -> 通过上岗
  -> 数据复盘
```

该闭环验证成功后，再逐步扩展到岗位培训和全公司前端到后端的AI提效。
