# 极享OS资产管理内置与交互重设计

> 日期：2026-07-03  
> 来源：设备资产管理系统知识库与当前极享OS代码结构梳理  
> 目标：把设备资产管理能力内置为极享OS一级业务模块，同时放弃独立系统里不再适合的抽屉式详情交互。

## 结论

资产管理这件事是对的，而且应该成为极享OS的一级业务域。

它不应该被放进系统设置，也不应该被当成普通配置表。设备、手机号、互联网账号都有归属、使用、风险、费用、审计和离职回收问题，会直接影响业务触达、安全和账号控制权。

但独立系统里的“右侧详情抽屉”不适合直接搬进极享OS。资产管理的核心动作是扫描台账、对比字段、追溯上下游关系、处理风险。抽屉会挤压表格宽度，并把关系追溯降级成附属信息。

## 保留的核心产品判断

一期仍然聚焦“核心资产台账闭环”，不扩成 OA、人事系统或大型风控平台。

主链路保持：

```text
设备资产 -> 手机号资产 -> 互联网账号资产
```

必须保留的能力：

- 设备、手机号、互联网账号三类资产独立维护。
- 设备详情能看到卡槽手机号和绑定账号。
- 手机号详情能看到所属设备和绑定账号。
- 账号详情能追溯绑定手机号和设备。
- 未绑定手机号的账号允许存在，但必须形成风险。
- 敏感字段查看必须独立授权并写审计日志。
- 离职回收必须服务于资产和账号权限回收，而不是泛人事流程。

## 极享OS内置形态

新增一级入口：

```text
/assets
```

侧边栏模块名：

```text
资产管理
```

页面工作区：

| 工作区 | 作用 |
| --- | --- |
| 资产总览 | 看资产规模、风险数量、离职待回收、费用与异常概况 |
| 设备资产 | 维护设备、IMEI、卡槽、归属、状态和风险 |
| 手机号资产 | 维护号码、运营商、所属设备、卡槽、套餐和费用 |
| 互联网账号 | 维护平台账号、绑定手机号、归属、权限状态和费用 |
| 风险提醒 | 处理未绑定、无负责人、离职待回收、异常状态等确定性风险 |
| 操作日志 | 查看新增、编辑、导入导出、敏感字段查看、回收等审计记录 |
| 离职回收 | 处理离职待回收账号和资产收回任务 |

## 新交互原则

### 1. 不使用右侧抽屉作为主详情

资产详情不再从右侧滑出。

原因：

- 资产表格字段多，抽屉展开后会显著挤压台账扫描区域。
- 用户需要在详情、关联资产、风险、日志之间来回判断，抽屉承载不够稳。
- 极享OS已有模块更偏全宽工作台，资产管理应保持一致。

### 2. 使用“全宽台账 + 页面内详情工作区”

资产列表保持全宽。点击资产行后，在同一页面内打开详情工作区，而不是侧滑抽屉。

推荐结构：

```text
模块头部
工作区页签
筛选与批量操作
资产台账表格
选中资产详情工作区
```

详情工作区包括：

- 当前资产摘要。
- 编辑资料入口。
- 上下游关联资产。
- 风险提示。
- 最近操作日志。

详情区可以出现在表格下方，也可以作为同页的“详情”子视图；关键是不要压缩主表格。

### 3. 关系追溯要成为主界面信息

关联资产不是备注字段。

设备详情应把卡槽手机号和绑定账号做成可点击入口；手机号详情应突出所属设备和绑定账号；互联网账号详情应突出绑定手机号和可追溯设备。

所有蓝色关联字段必须真的能跳转到对应资产上下文。

### 4. 表单仍然可以用弹窗

新增和编辑可以保留弹窗，但必须是正式后台表单，不是字段堆叠。

分组：

- 设备：基础信息、归属信息、状态与备注。
- 手机号：绑定关系、号码信息、费用与归属。
- 互联网账号：绑定关系、账号信息、归属与权限、状态与费用。

### 5. 风险与回收是资产工作台的一部分

风险提醒和离职回收不应该放到系统设置里。

风险是资产治理入口；离职回收是账号权限收回入口。它们都属于资产管理域。

## 权限颗粒度

建议新增权限：

- 资产管理
- 资产管理/资产总览
- 资产管理/设备资产
- 资产管理/手机号资产
- 资产管理/互联网账号
- 资产管理/风险提醒
- 资产管理/操作日志
- 资产管理/离职回收
- 资产管理/查看敏感字段
- 资产管理/导入导出

角色建议：

| 角色 | 建议权限 |
| --- | --- |
| 超级管理员 | 全部资产权限 |
| 老板 | 总览、风险、只读台账 |
| 资产管理员 | 设备、手机号、账号写权限，导入导出，离职回收 |
| 部门负责人 | 本部门资产只读与风险查看 |
| 审计员 | 操作日志、敏感字段查看记录、只读台账 |
| 普通员工 | 仅本人相关资产，只读且敏感字段脱敏 |

## 数据与 API 边界

核心实体：

- `AssetDevice`
- `AssetPhoneNumber`
- `AssetInternetAccount`
- `AssetRisk`
- `AssetOperationLog`
- `AssetOffboardingTask`

关键派生结构：

```text
AssetDetailBundle = 当前资产 + 关联手机号 + 关联账号 + 风险 + 最近操作日志
```

建议后端提供统一详情 API：

```text
GET /api/assets/:type/:id/detail
```

前端不要为了渲染一个详情区反复拼多张表。

## 实施顺序

### P0：模块壳和信息架构

- 新增 `/assets` 路由。
- 新增侧边栏“资产管理”入口。
- 新增资产权限 key。
- 新增 7 个工作区页签。
- 先用 mock 数据搭出全宽台账与页面内详情工作区，确认交互方向。

### P1：领域数据与行为

- 增加资产类型、mock 数据、API 封装和 store。
- 实现设备、手机号、互联网账号的查询、筛选、详情、编辑。
- 实现关系跳转。
- 实现未绑定手机号账号风险。

### P2：治理能力

- 敏感字段查看授权与审计。
- 风险解决/忽略。
- 离职回收任务。
- CSV 导入导出与失败行反馈。

### P3：后端持久化

- 将资产实体落入 Prisma/MySQL。
- 补唯一性、绑定关系、风险生成、敏感字段审计、离职回收的行为测试。

## 不做

- 不把资产管理放进系统设置。
- 不把审批流、人事流程做成主线。
- 不做复杂风控规则引擎。
- 不物理删除资产，优先停用、注销、回收。
- 不把独立项目的 React/Vite/Express 页面直接复制进极享OS。
- 不继续依赖右侧抽屉作为资产详情主交互。

## 下一步建议

下一步先实现 P0：只做极享OS里的资产管理模块壳、7 个工作区、全宽台账和页面内详情工作区。不要一开始就迁移完整后端，否则会把产品交互还没定稳的问题变成技术搬运问题。
