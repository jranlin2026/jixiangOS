# 业务附件、交付反馈与售后来源配置设计

日期：2026-07-16
状态：已完成业务方案确认，待书面复核

## 1. 背景与目标

员工验收暴露出三类问题：交付附件只保存文件名、交付步骤填写内容缺少明确保存反馈，以及订单和售后凭证只能保存一张且不支持剪贴板粘贴。同时，售后挽回需要复用客户/线索的联系方式规则，并把自由输入的来源平台升级为平台与店铺联动选项。

本批次目标：

1. 建立订单、售后挽回和交付共用的真实附件存储能力。
2. 让截图支持选择、拖拽、粘贴、预览和按上限追加。
3. 让交付步骤内容有独立保存入口、状态反馈和完成留痕。
4. 让售后挽回复用现有客户联系规则，并使用后台维护的平台/店铺选项。
5. 兼容现有订单和售后挽回中的单张内联图片，不破坏历史查看。

## 2. 范围

### 2.1 本批次包含

- 订单付款截图：最多 1 张。
- 订单成交路径/聊天记录截图：最多 8 张。
- 售后挽回收款凭证：最多 8 张。
- 售后挽回聊天记录：最多 8 张。
- 上述四类截图均支持文件选择、拖拽和剪贴板粘贴。
- 交付步骤附件真实上传、查看、下载、删除和重新上传。
- 交付步骤结果字段独立保存、未保存提示和保存成功反馈。
- 交付列表显示实际完成时间；已完成步骤显示完成时间和完成人。
- 售后挽回客户规则：客户名称必填，手机号或微信至少填写一项。
- 系统设置新增售后来源配置，维护平台及平台下的店铺。
- 售后挽回提交时先选平台，再选该平台下的店铺。

### 2.2 本批次不包含

- 按交付步骤配置不同业务字段的模板系统。
- 发票申请或发票审核流程。
- 对历史内联图片执行一次性离线迁移。
- OCR 同时识别多张付款截图；订单付款截图仍只有一张。

## 3. 方案选择

采用统一附件服务。真实文件由服务端存储，业务记录只保存附件元数据和受保护的附件标识。订单、售后和交付共享上传、读取与删除规则，界面共享多附件选择与粘贴逻辑。

不继续把新增图片以 Base64 写入业务 JSON。多张图片会放大请求体、数据库记录和列表查询成本；统一附件引用也能避免交付模块只保存文件名的假上传问题。

## 4. 附件领域模型

```ts
interface BusinessAttachment {
  id: string;
  name: string;
  mimeType: string;
  size: number;
  category:
    | 'order-payment-proof'
    | 'order-deal-evidence'
    | 'recovery-payment-proof'
    | 'recovery-chat-evidence'
    | 'delivery-task-file';
  uploadedById: string;
  uploadedByName: string;
  uploadedAt: string;
}
```

业务记录保存 `BusinessAttachment[]`，不保存磁盘绝对路径或公开 URL。查看和下载通过需要登录及业务权限校验的接口取得。原文件名只用于显示和下载响应，磁盘文件名由服务端生成，避免路径穿越和重名覆盖。

图片单文件最大 10 MB；交付文档单文件最大 20 MB。订单及售后截图仅接受常见图片格式。交付附件接受图片、PDF、Word 和 Excel 文件。服务端同时校验扩展名、MIME、数量和文件大小，前端校验只用于即时反馈。

## 5. 存储与权限

- 文件存放在独立的业务附件目录，不复用当前公开静态 `/uploads` 访问方式。
- 上传、查看、下载和删除均要求登录。
- 订单附件沿用订单查看/编辑权限和数据范围。
- 售后附件沿用售后挽回查看/编辑权限和数据范围。
- 交付附件沿用交付查看/写入权限和数据范围。
- 删除附件时先校验业务记录归属和编辑权限，再解除业务关联并删除物理文件。
- 数据库更新失败时清理刚写入的文件；物理删除失败时记录错误并返回失败，不伪装成功。

## 6. API 与数据流

统一上传接口接收 `multipart/form-data`，参数包含业务域、业务记录 ID、附件分类及文件。服务端不相信前端传入的所有权信息，而是根据业务记录和当前用户重新做权限判断。

上传成功后返回附件元数据。业务表单在提交业务记录时提交附件 ID 数组；服务端校验附件属于当前用户可操作的业务草稿或记录，并校验分类和数量上限。

附件下载使用附件 ID，服务端查出关联业务后鉴权并以 `Content-Disposition` 返回。图片预览使用同一受保护读取能力，不暴露磁盘目录。

删除接口以附件 ID 定位，不允许客户端传任意文件路径。已进入不可编辑状态的订单或售后挽回记录沿用现有锁定规则，不能通过附件接口绕过。

## 7. 历史数据兼容

读取订单和售后挽回记录时：

1. 优先读取新的附件数组。
2. 新数组为空且旧的文件名/预览字段存在时，将旧字段映射为只读的兼容附件。
3. 历史内联图片仍能预览，但未转存前没有独立附件 ID。
4. 用户编辑并保存该记录时，前端把历史内联图片上传到统一附件服务，再保存新附件数组。
5. 兼容期内详情、审核和分账页面同时支持新数组与旧字段，避免新建页正常但审核页丢图。

## 8. 前端附件交互

抽取共享附件选择逻辑和展示组件，统一处理：

- 文件选择支持 `multiple` 的区域可一次选择多张。
- 拖拽可一次加入多张。
- 区域获得焦点后，粘贴时读取剪贴板中的全部图片。
- 新文件追加到现有列表，不覆盖已有文件；同一批次按文件名、大小和最后修改时间去重。
- 达到上限时拒绝超出的文件，并说明当前数量、上限和被拒绝数量。
- 每张图片都有缩略图、文件名和单独删除按钮。
- 订单付款截图只保留 1 张；粘贴或拖入多张时拒绝整批并提示，不静默取第一张。
- 订单付款识别继续读取唯一的付款截图。

数量规则：

| 区域 | 上限 |
| --- | ---: |
| 订单付款截图 | 1 |
| 订单成交路径/聊天记录 | 8 |
| 售后收款凭证 | 8 |
| 售后聊天记录 | 8 |

## 9. 交付改进

### 9.1 步骤内容保存

每个交付步骤提供独立“保存”按钮。字段发生变化后显示“未保存”；保存中禁用重复提交；成功后显示“已保存”和最近保存时间；失败时保留输入内容并显示错误。

勾选完成步骤时仍会连同当前结果字段一起保存。右侧“保存资料”只负责交付资料，并调整说明文字，避免员工误认为它会保存所有步骤。

关闭存在未保存步骤内容的交付详情前进行确认。不同步骤分别保存，单个步骤失败不影响其他步骤。

### 9.2 附件

交付附件不设置本批次截图的 8 张业务上限；仍受单文件大小和允许类型限制。附件上传成功后显示真实可操作条目：图片预览，其他文件查看名称和下载；有写权限的用户可以删除并重新上传。

### 9.3 完成时间

- 列表增加“实际完成时间”列，取交付主管确认完成产生的 `actualCompletedAt`。
- 未完成时显示 `-`，不使用更新时间冒充完成时间。
- 每个已完成步骤显示 `completedAt` 和 `completedBy`。
- 计划完成时间继续独立展示，不与实际完成时间互相覆盖。

## 10. 售后挽回规则

### 10.1 客户联系方式

复用客户/线索新增规则：

- 客户名称必填。
- 手机号或微信至少填写一项。
- 前后端使用同一规则，提示文案为“手机号或微信至少填写一项”。
- 手机号和微信继续参与客户查找与重复识别；不把客户名称与微信改成二选一。

### 10.2 平台与店铺配置

新增配置模型：平台包含稳定 ID、名称、启用状态、排序；店铺隶属于平台，也包含稳定 ID、名称、启用状态和排序。

系统设置新增“售后设置 / 来源平台与店铺”页面，并新增独立权限 `系统设置/售后设置/来源平台与店铺`。只有具备该权限的用户可以维护配置。

售后挽回表单：

- 来源平台为必选下拉框。
- 选择平台后，只展示该平台下启用的店铺。
- 店铺为必选。
- 切换平台时清空不属于新平台的店铺选择。
- 业务记录同时保存平台/店铺稳定 ID 和名称快照，配置改名不重写历史记录。
- 停用选项不能用于新建，但编辑历史记录时仍能显示原值。

## 11. 错误处理

- 单个附件上传失败不会清空已经成功的附件；界面标出失败文件并允许重试。
- 业务表单提交失败时保留已上传附件引用和本地表单内容。
- 用户取消尚未提交的新增表单时，服务端清理未关联的临时附件；定期任务兜底清理过期临时附件。
- 附件读取时记录不存在返回 404，无权限返回 403，不泄露文件是否存在于磁盘。
- 平台或店铺在填写期间被停用时，提交返回明确冲突提示，用户重新选择后再提交。

## 12. 测试与验收

开发按测试先行执行，覆盖：

1. 各区域数量上限；订单付款多文件整批拒绝。
2. 多选、拖拽和粘贴追加，删除后可继续添加。
3. 前端校验和服务端绕过前端后的再次校验。
4. 上传、下载、预览和删除的权限及数据范围。
5. 历史单张 Base64 数据在新增页、详情页、审核页和分账页可见。
6. 交付附件保存真实内容，不能再出现只有文件名的记录。
7. 步骤独立保存、失败保留输入、未保存关闭提示。
8. 实际完成时间与步骤完成留痕。
9. 售后客户名称必填，以及手机号/微信二选一。
10. 平台与店铺联动、停用处理和历史快照。
11. 类型检查、全量测试、生产构建和针对上传接口的集成测试。

## 13. 发布与回滚

- 新字段均为向后兼容的可选字段，旧字段在兼容期保留。
- 发布前确认生产附件目录持久化、容量、备份和进程写权限。
- 先部署支持新旧结构的服务端，再部署前端。
- 回滚前端时服务端仍保留兼容读取；回滚服务端前不得产生旧版本无法理解的新附件数据。
- 发布后用订单、售后和交付三个角色各执行一次上传、查看、下载、删除冒烟验证。
