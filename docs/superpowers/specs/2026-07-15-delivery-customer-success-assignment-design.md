# 交付中心客户成功顺序分配设计

## 背景

交付单目前会优先读取订单上的客户成功或售后人员；订单未携带这些字段时，交付负责人进入“待分配”，需要主管逐单手工处理。手工分配保存后还会复用详情刷新函数，导致分配窗口关闭后自动打开整张交付详情。

本设计实现用户确认的规则：所有产品共用一个客户成功人员池，严格按照参与人员的配置顺序循环分配，不按产品和工作量分池。手工分配保存后只刷新列表，不再自动打开详情。

## 目标

- 所有新建交付单共用一个客户成功人员池。
- 按配置顺序循环分配，例如 A → B → C → A。
- 分配游标持久化，服务重启后继续上一次位置。
- 自动跳过已暂停、停用或离职人员。
- 无可用人员时保留“待分配”，不得阻断交付单创建。
- 主管可以手工改派，人工改派不改变自动轮询游标。
- 手工分配成功后关闭分配窗口、刷新列表并反馈成功，不打开交付详情。

## 非目标

- 不按产品、产品等级、区域或行业建立多个人员池。
- 不按进行中交付数量或优先级计算工作量。
- 不自动重分配已有交付单。
- 不在本次范围接入短信、邮件或第三方消息通知。

## 设置入口与权限

在“系统设置”新增“交付设置”分组，并新增“客户成功分配”页签：

`系统设置 → 交付设置 → 客户成功分配`

新增独立权限键：`系统设置/交付设置/客户成功分配`。默认仅超级管理员和具备该权限的客户成功主管可以查看、保存配置。交付中心现有分配权限继续控制人工改派，两种权限互不替代。

## 配置界面

页面包含：

1. “启用自动分配”开关。
2. 有序参与人员列表，支持添加、移除和拖动排序。
3. 每位参与人员的“暂停接单/恢复接单”状态。
4. “下一位预计分配人员”只读提示。
5. 保存按钮及保存成功、保存失败反馈。

候选员工来自在职且启用的员工目录。配置中保留但已经停用或离职的员工应显示异常状态，并在运行时自动跳过，便于管理员清理配置。

## 数据模型

配置使用服务端持久化，结构如下：

```ts
interface DeliveryAssignmentConfig {
  enabled: boolean;
  participants: Array<{
    userId: string;
    paused: boolean;
  }>;
  lastAssignedUserId?: string;
  updatedAt: string;
  updatedBy: string;
}
```

`participants` 数组顺序就是分配顺序。`lastAssignedUserId` 是持久化游标；当成员被删除或顺序变化后，从新列表中第一个有效成员继续。

交付单记录自动或人工分配的结果：

- `ownerId`、`owner`：当前负责人。
- `assignmentMode`：`auto` 或 `manual`。
- `assignedAt`：最后分配时间。
- `assignedBy`：人工改派时记录操作人；自动分配记录 `system`。

## 自动分配算法

自动分配在交付单创建事务内执行：

1. 自动分配关闭时，负责人保持订单已有负责人；没有则为“待分配”。
2. 自动分配开启时，读取配置并锁定游标，避免并发订单分给同一个位置。
3. 从 `lastAssignedUserId` 的下一位开始，按数组顺序循环查找。
4. 跳过暂停接单、停用、离职或不存在的人员。
5. 找到候选人后写入交付负责人，同时把 `lastAssignedUserId` 更新为该人员。
6. 没有有效候选人时创建交付单但负责人为“待分配”，游标不变。

自动分配覆盖所有正常创建入口，包括订单审核自动建单和交付中心手工补建。人工改派只更新交付单，不更新 `lastAssignedUserId`。

## 并发与失败处理

- 配置读取、候选人选择、游标更新和交付单创建必须处于同一数据库事务。
- 同一时刻并发创建多张交付单时，每张交付单依次消耗一个轮询位置。
- 配置损坏、人员池为空或没有有效成员时，降级为“待分配”，但记录明确的系统结果，不伪报自动分配成功。
- 配置保存时拒绝重复人员 ID，并校验操作者权限。

## 手工分配交互修复

保存人工分配成功后：

1. 关闭“分配交付负责人”窗口。
2. 刷新交付列表和统计。
3. 显示“分配成功”反馈。
4. 保持 `selectedDelivery` 为空，不请求和打开交付详情。

只有点击“查看交付”按钮时才打开完整详情。

人工分配下拉框只展示在职、启用并且属于配置人员池的成员；自动分配未配置时，可由具备权限的主管从符合客户成功岗位范围的在职人员中选择，不能继续展示全公司所有员工。

## 服务接口

- `GET /api/settings/delivery-assignment`：读取配置和解析后的参与人员状态。
- `PUT /api/settings/delivery-assignment`：保存开关、人员顺序和暂停状态。
- 现有交付更新接口继续承担人工改派，并补充稳定的分配审计字段。

所有设置接口要求“客户成功分配”设置权限；交付更新接口继续要求交付写权限和数据范围校验。

## 验收标准

1. 配置 A、B、C 后连续创建四张交付单，负责人依次为 A、B、C、A。
2. B 暂停后连续创建交付单，只在 A、C 之间循环；B 恢复后重新参与后续轮询。
3. 服务重启后下一张交付单从持久化游标的下一位开始。
4. 两张并发订单不会获得同一个轮询位置，除非人员池只有一人。
5. 人员池无有效成员时交付单仍创建成功，负责人为“待分配”。
6. 人工改派后下一张自动分配仍沿用原轮询游标。
7. 人工分配保存后不打开交付详情，列表立即显示新负责人。
8. 无设置权限的账号不能读取或修改分配配置。

## 测试范围

- 顺序轮询、循环、暂停、停用、离职、空池和游标恢复单元测试。
- 并发创建和事务游标集成测试。
- 设置接口权限与输入校验测试。
- 订单审核自动建单、手工补建和人工改派回归测试。
- 设置页面成员排序与交付页面“分配后不打开详情”静态或交互测试。

## 回滚

关闭“启用自动分配”即可立即停止新的自动分配。代码回滚不删除已保存配置；已有交付负责人保持不变，后续继续允许主管人工改派。
